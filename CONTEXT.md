# CONTEXT.md — 用語の正本

> ドメイン用語のブレは仕様のブレになる。ここに無い用語で仕様表・テスト・コードを書かない。
> ズレを見つけたら **先にここを直してから**成果物を直す（`/flywheel`・grill-with-docs の対象）。
> 由来: `docs/必要画面・機能一覧.md`・`docs/画面仕様/`・README.md。

## アクター

| 用語 | 英語（コード上） | 意味 |
|---|---|---|
| エンジニア | engineer | 業務報告を入力し、スキルシート・推奨案件の提示を受ける利用者 |
| 営業担当 | sales | エンジニアを管理し、スキルシートを確定し、推奨案件の開示可否を判定する利用者 |

## ドメイン用語

| 用語 | 英語（コード上） | 意味 |
|---|---|---|
| 業務報告（日報） | report | エンジニアが入力する日々の報告。AI が整形し、本人が確認・修正して保存する |
| AI 整形 | (Summarizer 経由) | 自由記述/アンケート形式の入力を構造化された報告に変換する処理 |
| スキルシート | skill sheet | 蓄積された業務報告から生成される、エンジニアのスキル一覧文書。営業担当が編集・確定する |
| 推奨案件 | recommended project | エンジニアの希望に基づき生成され、営業担当の開示判定を経て提示される案件 |
| 開示判定 | disclosure decision | 営業担当が推奨案件をエンジニアに提示してよいか判断する行為（開示 or 拒否） |
| Excel テンプレート | excel template | スキルシート出力に使うテンプレート。営業担当が管理する |

## 型（方法論）の用語 — M-team から継承

| 用語 | 意味 |
|---|---|
| answer key | 仕様の正本。このリポジトリでは `docs/画面仕様/`＋`docs/必要画面・機能一覧.md`（read-only） |
| 受け入れテスト | answer key を翻訳したテスト＝仕様。API 層 `*.integration.test.ts` ＋ UI 層 `*.acceptance.test.tsx`。read-only（ADR-0001） |
| スライス | 1 issue = 1 セッションで完了する作業単位。採番は `/board`（ADR-0013・不変） |
| 仕様表 | `docs/spec/slice-NN.md`。grill の成果物。`approved: true` が Phase B の関所（ADR-0012） |
| スライス指示書 | `docs/slices/slice-NN.md`。6項目。正本は repo、issue はポインタ（ADR-0006） |
| 層境ゲート | PM の GO/NO-GO（工程8）。緑 ≠ 仕様充足を人が判定する関所 |
| 統合役 | main への不可逆操作（マージ・マイグレーション）を行う唯一の人 |
| 救援 | 下流がリーダーへ質問として出すこと。AFK 未完走にカウント（恥ではなく計測対象） |
| fix-forward | 総合テスト赤でも revert せず、回帰スライスを append して前へ進む（ADR-0014） |
| L0/L1/L2 | 合成フィクスチャの機密階層。L0=完全合成 / L1=構造のみ実物由来 / L2=要マスキング（dev は L0 のみ） |

## 表記ゆれの禁止

- 「日報」「業務報告」→ コード・テスト名は `report` に統一（画面表示文言は画面仕様に従う）
- 「案件」単体ではなく「推奨案件」（recommended project）を使う（開示判定前後で状態が違うだけで実体は同じ）
- ステータスコードの根拠は「仕様（画面仕様・受入基準）」か「実装規約（ErrorKind）」かを常に区別する
