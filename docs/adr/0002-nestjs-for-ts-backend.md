# TS バックエンドは NestJS を採用する（Hono は不採用）

> **2026-07-09 追記：本 ADR は [ADR-0011](./0011-express-with-enforced-layering.md) に置き換えられた。**
> 採用フレームワークは Express になったが、**判断の核心（構造が Spring と 1:1 であること）は維持されている**。
> 変わったのは強制の担い手で、フレームワークからハーネス（規約＋カスタム lint）へ移した。以下は当時の記録。

初期 TS 実装のバックエンドフレームワークとして NestJS を採用する。決め手は最終形が Java/Spring Boot であること：NestJS の構造（DI・デコレータ・Module/Controller/Service 分離）は Spring Boot とほぼ 1:1 対応し、TS→Java 移行を「アーキテクチャ翻訳」でなく「構文翻訳」に近づける。また下流メンバーが TS 期間中に見る構造がそのまま Spring の予習になり、成功体験（C）の質も上がる。

## 不採用の理由（Hono）

Hono は軽量で AI が最速に書ける（B優先の観点では優位）が、本プロジェクトは B を主目的にしない（憲法 §1「A主・C従」）。受け入れテストはブラックボックス専用（ADR-0001）なのでフレームワーク差はテスト資産に影響せず、選定は「移行のしやすさ」を最優先とした。NestJS のボイラープレートの多さは Claude Code が定型生成を担うため実質的なコストにならない。
