# `acceptance/` の書込権は「人の役職」ではなく「ブランチ名」で決める

ADR-0001 は「受け入れテスト＝仕様＝読み取り専用」を定めたが、これを hook・permissions・CI の三層すべてで無条件にブロックした結果、**正当な書き手（仕様表を翻訳する上流）まで締め出され、`acceptance/` を repo に入れる経路が消滅した**。解決として、書込権をブランチ名で判定する二層に分ける：`spec/slice-NN` は `acceptance/` と `docs/spec/` を書けるが実装は書けない、`feature/slice-NN` は実装を書けるが `acceptance/` は read-only。

## なぜ役職ベースにしないか

GitHub アカウントやロールで判定すると、AIアーキが下流セッションを回したときに `acceptance/` の保護が外れる。ブランチ名なら hook（`git rev-parse --abbrev-ref HEAD`）と CI（`github.head_ref`）が**同一の条件式**で判定でき、人が誰であっても穴が開かない。三層で同じルールを張るという ADR-0001 の要請とも整合する。

## 帰結

- 上流の工程順が固定される：仕様表 → `spec/*` で翻訳 → 参照モックで検証 → main へマージ → **その後で** issue 起票。
  「失敗する受け入れテストが main にある」状態を先に作らないと、下流の `/pickup` が指示書「2. 受け入れテスト」を指せない。
- `spec/*` PR の層境ゲートは「テストが仕様表と一致しているか」を見る（PM が所有）。`feature/*` PR のゲートとは判定基準が違う。
