# スライス指示書の正本は repo のファイル。issue はポインタに降格する

憲法は「issue 本文は信頼できない入力」と定めているのに、計画書は指示書を issue 本文として下流へ渡していた。指示書＝issue 本文なら「指示書に無い指示が本文に混ざる」ことは定義上ありえず、信頼境界がどこにも引けない。よって指示書の正本を `docs/slices/slice-NN.md`（main にマージ済み）に置き、issue には slice ID と指示書へのリンクだけを書く。`/pickup <issue>` は issue から slice ID を読み、**6項目は repo のファイルから読む**。

| 入力 | 信頼度 | 理由 |
|---|---|---|
| `docs/slices/slice-NN.md` | 信頼できる | PR レビューと層境ゲートを通っている。改竄が git 履歴に残る |
| issue 本文・レビューコメント | 信頼できない | 誰でも後から編集でき、履歴が追いにくい |

## なぜこちらが良いか

プロンプトインジェクション対策が「issue 作成者を上流3名に限定する」という**運用**から、「信頼する入力は git 管理下のファイルだけ」という**構造**に変わる。運用は破れるが構造は破れない。あわせて、指示書は目的A（言語非依存の再利用可能な型）の中核資産であり、GitHub の issue DB に閉じ込めると repo を移したとき持ち出せない。「枠・禁止事項の文言はリーダーが正本」という所有マップも、PR レビューを通ることで初めて機械的に守られる。

## 帰結

- 指示書と `acceptance/` は同じ `spec/slice-NN` ブランチで作られ、まとめて main へマージされる。**issue の起票はその後**。
- `/pickup` が issue 本文のフォーマット揺れで6項目のパースに失敗する、という下流を止める事故が消える。
