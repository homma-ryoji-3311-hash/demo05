# 層境ゲートは全 PR に掛け、重さを2段に分ける。判定者は PM（代理はリーダー1名）

計画書は「ゲートは不可逆変更のみ」（§2）と「ゲートは全スライス」（§9）の二枚舌だった。前者では大半の PR で「緑 ≠ 仕様充足」の判定者が消え、後者ではリーダーが全 diff を読む人間ボトルネックになる。よって **全 PR にゲートを掛け、重さを2段に分ける**。判定者は `spec/*`・`feature/*` とも **PM**（当初計画のリーダーから変更）。

| ゲート | 判定者がやること | 対象 |
|---|---|---|
| **軽量** | Audit の推奨判定と統合役の再検証結果を読んで GO/NO-GO。diff は自分で読まない | それ以外 |
| **重量** | PM が diff を自分で読む。Audit の GO を鵜呑みにしない | `irreversible` ラベル付き PR |

`irreversible` ラベルは **CI が機械的に貼る**（`prisma/schema.prisma` / `**/migrations/**` / 認可コード / `acceptance/**` に触れる PR）。「重要そうだから丁寧に見よう」という運用は必ず破れる。

## なぜこの形か

hooks が「ブロックは不可逆・高コスト操作に集中投下」であるのと**同じ原則をゲートに適用しただけ**で、憲法に新しい概念が増えない。Audit の判定は推奨にすぎず PM を拘束しない（`GO` でも PM は NO-GO を出せる）。PM が仕様表を書き、ゲートで評価するのは AI が翻訳したテストと下流が書いた実装なので、自己レビューにはならない。

## 帰結

- リーダーは関所を持たなくなる。責務は「下流の一次窓口」「枠・禁止事項の文言」「救援の記録」の3つ。
- **代理はリーダー1名に限り、全ゲートで認める**（重量ゲートも含む）。これは「仕様が PM の承認なしに main に入りうる」ことを意味するため、代理判定は `docs/metrics/gates.md` に `代理: リーダー` として記録し、PM が事後に確認する。
- AFK 完走率（緑＋PR まで）はゲートの重さに影響されない。
