# 「画面が参照モックと一致」は golden スクリーンショットで機械判定する

> **前提が崩れている（2026-07-15・ADR-0018 で一部補完）。** 本 ADR は「参照モックに撮影できる画面がある」
> ことを前提に書かれたが、現物の参照モックは**文書ベースで撮る画面が無かった**。本 ADR にはこの場合の
> 縮退経路が無く、工程4 の翻訳者が「pixel 比較できない」→「UI 検証不要」と読み替える余地を残した。
> **撮影可否と UI 検証の要否は別の判断である**（ADR-0018）。撮れないときは role/label ベースの
> DOM アサーションへ縮退する。下の「DOM の構造アサーションのみ」の却下は、**golden が撮れる状況での
> 比較検討**であって、撮れない状況での代替を否定したものではない。

憲法は「初級者が担う判定は3つだけ（**知識ゼロで○×できるもの**）」と定めているのに、「画面がモック通り」だけは人が目で見比べる主観判定だった。ADR-0005 で参照モックを repo に vendor した結果、両方を同時に起動できるようになったので、参照モックの画面を **golden スクリーンショットとして `acceptance/golden/` にコミット**し、Playwright の `toHaveScreenshot()` で pixel 比較する。初級者の判定は「赤い数字が出ていないか」だけになる。

golden を撮るのは **`spec/*` ブランチの工程**（上流）。実装より先に撮る——後から撮ると**実装が仕様を定義する**ことになり、順序が逆転する。

## なぜ golden を `acceptance/` に置くか

`acceptance/` の read-only 保護（宣言・hook・CI の三層）がそのまま効き、「見た目が違うから golden を撮り直す」逃げ道が塞がれる。これは**テストを書き換えて緑にするのと同じ不正**なので、同じ強制力で防ぐべきものである。画面（Next.js）は Java/Spring 移行の対象外なので、golden はバックエンドを差し替えても不変の資産になる。

## 帰結（導入時に必ず要るもの）

pixel 比較はフォントレンダリング・OS・ブラウザ版で簡単に割れ、「同じテストを3回リトライして緑にならない＝ハーネスのバグ」という自動停止トリガーに真っ先に引っかかる。したがって：

- CI とローカルで**同一のブラウザ版**に固定する（Playwright のバージョン pin、または公式 Docker イメージ）
- 閾値は最初ゆるく（`maxDiffPixelRatio: 0.02` 程度）、Flywheel で締める
- アニメーション・日時・ランダムIDは `mask` するか、合成フィクスチャで固定する

## 考慮した代替案

- **DOM の構造アサーションのみ**：「参照モックと一致」が保証されない。動くが見た目が違うものを「完成」と呼ぶことになり、目的C（成功体験）の質が落ちる。
- **視覚比較を CI 専用にする**：下流が「緑だと思って PR → CI で赤」を繰り返し、AFK 完走率が下がる。フィードバックは左に寄せる。
