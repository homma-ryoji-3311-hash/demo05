# 上流にも一本道を敷く（`/spec` と `/brief`）。工程とブランチの対応は機械で縛る

型が下流にしかないなら半分の型でしかない。ただし上流の全工程をコマンド化はしない。**コマンド化の基準は「繰り返す × 手順が固定 × 間違うと痛い」の3条件**とし、満たすものだけ作る。

| 工程 | コマンド | 理由 |
|---|---|---|
| 基本設計 | **作らない** | MVP 全体で1回きりの判断業務。素の会話＋`grill-with-docs` |
| 仕様表 → `acceptance/` 翻訳＋golden 撮影 | **`/spec <slice>`** | 毎回・固定・`acceptance/` を触る唯一の場所 |
| スライス指示書＋issue 起票 | **`/brief <slice>`** | 毎回・固定・6項目の欠落は下流を止める |
| issue 分解 | **作らない** | 1回きり。既製の `to-issues` で足りる |

## `/spec` の存在理由

内部で `reference-mock` に対してスイートを流して**緑**を確認し（翻訳が正しい証明）、続いて `backend` に対して流して**赤**を確認する（下流に渡せる状態の証明）。この**緑→赤の反転**を人が手でやると必ずどちらかを飛ばし、しかも「テストが赤いのは正常」なので誰も気づかない。上流で最も価値のある自動化はここ。

## 工程とブランチの機械的対応

判断業務は高自由度、壊れやすい操作は低自由度、という skills 設計規範に従い、次を hook で強制する。

- `/spec` を `feature/*` で叩いたら停止（`acceptance/` を実装ブランチから触らせない）
- `/implement` を `spec/*` で叩いたら停止（実装が spec ブランチに紛れ込むのを防ぐ）

ADR-0004 でブランチ名を書込権の判定軸にした以上、工程の判定軸も同じであるべきで、条件式を1つに保てる。
