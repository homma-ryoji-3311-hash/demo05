# `/spec` は仕様表の作成も担う。ただし2フェーズに割り、間に PM の承認を挟む

仕様表が無ければ `/spec` は停止する、という設計は上流の一本道を途切れさせていた。よって `/spec <slice>` に**仕様表の作成（Phase A）**を取り込み、**叩くのは PM 自身**とする。仕様表が無い（または `--regrill` 指定）とき、`grill-with-docs` で PM に質問しながら `docs/spec/slice-NN.md` を書く。

**Phase A と Phase B の間で必ず停止する。** `docs/spec/slice-NN.md` の frontmatter に `approved: true` が書かれるまで、Phase B（`acceptance/` への翻訳）に進めない。これは hook（`protect-paths.sh`）で機械的に強制する。

| | Phase A：仕様表 | Phase B：翻訳 |
|---|---|---|
| 性質 | **判断業務**（高自由度） | **壊れやすい操作**（低自由度・順序固定） |
| 主体 | PM が答え、AI が書く | AI が実行 |
| 出力 | `docs/spec/slice-NN.md` | `acceptance/` ＋ `golden/` ＋ PR |

ひと続きで走らせると、AI が自分で書いた仕様表を自分でテストに翻訳して自分で PR を出す。**生成と評価の分離**が消える。

## なぜ PM が叩くか

grill に答えられるのは要件を知っている人だけである。AIアーキが代理で答えると、**仕様の所有が静かに AIアーキへ移る**。「上流はコードを書かない（が、AI は動かす）」の原則は PM にも等しく適用される。

> **注記（ADR-0017 で更新）**：この「PM が叩く」の限定は、PM と AIアーキが**別人格のときの所有ドリフト**を防ぐためのものだった。単一アカウント運用（ADR-0015）では両者は同一人物であり、identity 分離は元々成立しない。よって ADR-0017 は**キックオフ＋上流に限り**、`/spec` を叩く主体・grill に答える主体・`approved: true` を押す主体を **PM／AIアーキ同列**に開いた。ただし本節が守っていた実質——「grill の答えは人間から取る（AI が自問自答しない）」「各受入基準に `source:` を記録する」「Phase A/B を停止で割る（`protect-paths.sh`）」——は ADR-0017 でも一切変更しない。所有ドリフトの懸念は「別人格の PM がいる」多アカウント化まで再訪する。

## 循環参照への対策

AI が仕様表を書く際、最も参照しやすい材料は `reference-mock/` そのものである。放置すると

```
参照モック →(AIが読む)→ 仕様表 →(AIが翻訳)→ テスト →(参照モックに流す)→ 当然、緑
```

となり、Phase B の「参照モックで緑」が**自己証明**に退化する。「テストが仕様表と一致する」ことは検証できるが、「仕様表が正しい」ことは何も検証していない。対策：

1. **grill の答えは必ず PM から取る。** AI がモックを読んで下書きを作るのは可。しかし「このスライスに含める／含めない」「この 422 は仕様か実装都合か」は PM が答える。**AI が自問自答してはならない。**
2. **各受入基準に `source:` を記録する。** `reference-mock`（AI 下書き）か `PM`（新規決定）か。**参照モックに無い項目＝本当の設計**であり、そこだけ PM が重点的に見ればよい（工程1 の「書き起こしと設計を混ぜるな」と同型）。

## 帰結

- 工程3（仕様表）と工程4（翻訳）は `/spec` の Phase A / Phase B になる。工程は消えず、コマンドが1本になる。
- `protect-paths.sh` はブランチ名で書込権を判定し（ADR-0004）、さらに **`spec/*` であっても `approved: true` が無ければ `acceptance/` への書込をブロック**する。承認は口頭でなく git に残る。
- PM が Claude Code を叩ける必要がある。これは目的A（人がAIを動かす型）を上流にも適用するということであり、退行ではない。
