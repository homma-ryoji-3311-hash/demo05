# スライスの採番と一覧は `/board` が単独で所有し、番号は不変・append-only とする

issue 分解（工程2）で切り出したスライスに番号を振り、一覧表で管理する担い手が誰にもいなかった。`docs/slices/README.md`（スライスレジストリ）は playbook が「工程2 の成果物」と書いていたが実体が無く、番号を採る仕組みも「今どのスライスがどの工程か」を見る場所も存在しなかった。よって**スライス管理専用のスキル `/board` を1本**新設し、採番・一覧表・現在地表示をここに集約する。

## 決めたこと

1. **`/board` が `docs/slices/README.md`（スライスレジストリ）を単独で所有する。** 採番と一覧表の生成・更新はここだけで行う。他スキルはこの表を**読むだけ**。
2. **番号は不変（never renumber）・append-only。** 工程2 で `to-issues` の縦切り分解と `docs/design/overview.md` を読み、依存順に `slice-01..NN` を一括採番する。以後この整数は二度と振り直さない。スライスの分割（工程3 の受入基準6個超）や回帰（ADR-0014）で新スライスが生じたら、**末尾に append** する。
3. **「仮番号」の概念を置かない。** 工程2 の採番がそのまま確定番号。
4. **状態は git / ファイル / gh から推論する（状態を書き込まない）。** 各スライスが「今どの工程か」は、`spec/slice-NN` ブランチの有無・`docs/spec/slice-NN.md` の `approved: true`・`docs/slices/slice-NN.md`（指示書）の有無・PR / マージ状態（gh）から `/board` が読み取る。既存スキルにステータス書き込み処理を足さない。
5. **各工程の入口で `/board` を叩いて現在地を確認する。** `prime` と同じ「実態から現在地を読む」read-only オリエンテーションとして機能する。`disable-model-invocation: true`（人が明示的に叩く）。箱＝AIアーキ。

## なぜ番号を不変にするか（振り直しは参照を壊す）

`slice-NN` の `NN` は多数の**永続的な場所**に焼き込まれる：ブランチ名（`spec/slice-NN` / `feature/slice-NN`）、`acceptance/` のパス、`docs/spec/slice-NN.md`、`acceptance/golden/` のファイル名、issue のポインタ。

検討の過程で「工程2 は仮番号にして `/brief`（工程5）で詰め直す」案が出たが、これは成立しない。**`/brief` は `/spec`（工程3・4）より後**に走る（playbook の順序制約：失敗する受け入れテストを main に置いてから issue を配る）。`/brief` の時点で `NN` は既に spec ブランチ・仕様表・acceptance パス・golden 名に焼き込まれており、そこで振り直すと全参照が孤児になる。よって**採番したら不変**とし、変更はすべて append で吸収する。

## なぜ状態を書き込まず推論するか

各スキル（`/spec`・`/brief`・`/submit`）が完了時にレジストリの状態列を更新する「決定論的更新」案もあった。`docs/metrics/slices.md` への KPI 追記という先例もある。しかし状態を二重に持つと必ずドリフトする（更新漏れ・巻き戻し時の不整合）。git ブランチ・`approved: true`・ファイルの有無・PR 状態は**それ自体が動かしようのない実態**であり、そこから読めば状態は嘘をつかない。M-team の「自己申告に頼らない・生出力で確認」（CLAUDE.md）と同型であり、既存スキルへの結合も生まない。

## 帰結

- 工程2 は `to-issues`（縦切り分解）→ `/board`（採番・表生成）の順になる。`to-issues` は番号を持たない生の一覧を作り、`/board` が不変番号を与える。

> 訂正（2026-07-13）：本 ADR 起草時の「to-tickets」表記は計画書の誤記由来。vendor 済みスキルの実名は `to-issues`（PROVENANCE.md 参照）のため表記を修正した。決定内容に変更なし。
- レジストリの一覧表と、1スライスを下流へ渡す**スライス指示書**（`docs/slices/slice-NN.md`）は別物。前者は全スライスの索引、後者は1スライスの受け渡しパケット。CONTEXT.md で語を分ける。
- ADR-0014 の回帰スライスは `/board` の append で採番される。「不変・append-only」は fix-forward 運用の前提でもある。
