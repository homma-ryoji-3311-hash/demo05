# 統合役の再検証〜マージ〜総合テストを `/integrate`（二フェーズ）に束ね、マージ後 E2E の赤は fix-forward で回帰する

工程7（統合役の再検証）→ 工程8（PM の GO/NO-GO）→ 工程9（マージ）は、手作業のまま残っていた唯一の連続工程だった。さらに「個別に緑な2つの PR がマージ後に赤」（ADR-0010 が構造的に検出不能と名指しした事象）を捕まえる総合テストの実行場所が、CI（Step3）まで空白だった。よって統合役の工程を `/integrate` に束ね、マージ直後に総合テスト（E2E / Playwright）を走らせ、赤なら回帰スライスで前に進んで直す。

## 決めたこと

1. **`/integrate` は統合役ただ1人が叩く二フェーズスキル。** `disable-model-invocation: true`。`/spec` と同じく、間に人の判断（工程8 の PM GO）を挟むために停止する。
   - **Phase A**：工程7 の再検証（当該スライスのスイート＋シークレット/PII＋差分の目視）を実行し、結果を出して**停止**する。ここで PM の GO/NO-GO（工程8・GitHub 上）を待つ。
   - **Phase B**（GO 後に再実行）：工程9 のマージ → 直後に工程9b の総合テスト。
2. **マージは `gh pr merge`（GitHub の server-side マージ）で行う。** ローカル `main` への push はしない。main 防御の正本は GitHub ブランチ保護（マージ権限は統合役のみ）であり、`/integrate` はその権限の下で mechanical steps を代行するだけ。**hook もブランチ保護も迂回しない。**
3. **総合テストはマージ後に main で走らせる（工程9b・新設）。** `reference-mock` ＋ `backend` ＋ `frontend` を起動した実 HTTP / Playwright E2E の全スイート。
4. **総合テストの赤は fix-forward で扱う。** マージ後に赤が出れば main は一時的に赤になりうる。巻き戻さず、回帰スライスを `/board` で append 採番して前に進んで直す。これは ADR-0010 が「CI が捕まえる」と想定した事象を、CI 導入前から手元で捕まえる位置づけ。
5. **回帰ループの入口は分析で2分岐する。**
   - **既存の総合 E2E が赤（テストは在った）** → 工程5 `/brief` で回帰スライスを起票 → 工程6 で緑化。
   - **総合スイートに穴（この回帰を捕まえるテストが無い）** → 工程3 `/spec` で回帰テストを先に作る → 4 → 5 → 6。
   どちらも新スライスは `/board` の append で採番。分析は `/flywheel` の2分類（スライス設計の欠陥／ハーネスの欠陥）を流用し、`docs/metrics/` に記録する。

## なぜマージをスキル化してよいか（不可逆操作の原則を破らない）

CLAUDE.md §1 の絶対禁止「main を進める操作は統合役ただ1人」は維持される。`/integrate` の不可逆部分（Phase B）は**人の GO（工程8）の後にしか走らない**。マージ実行は `gh pr merge`＝GitHub 権限で server-side に行われ、ローカル push を伴わないため hook のブランチ判定にもブランチ保護にも触れない。`/submit` が PR 作成を代行するのと同じ立て付けで、判断は人・機械的手順だけをスキルが担う。「勢いを承認と誤解する」失敗を避けるため、`disable-model-invocation: true` で自動発火を封じる。

## なぜマージ後に総合テストを置くか（main 一時赤の許容）

マージ前に一時ブランチで E2E して緑なら fast-forward する「マージキュー型」なら main は常に緑に保てるが、統合役の手順が重く gh 標準フローから外れる。M-team の規模と、`/board` の append + 回帰ループが既に fix-forward を前提にしていることから、**マージ後 E2E ＋ main 一時赤 ＋ 回帰スライスで前進**を選ぶ。これは ADR-0010 が「マージ後に赤」を正常に扱う想定と整合する。Step3 で CI on main が自動化を引き継ぎ、将来 required check としてマージ前に前寄せする余地は残す。

## なぜ回帰ループを2分岐にするか

方法論の不変条件は「下流は既にある失敗する受け入れテストを緑にするだけ・`acceptance/` は書けない」。回帰を直すにも、まずその回帰を捕まえる失敗するテストが要る。既存の総合 E2E が赤なら「テストは既にある」ので工程5 へ直行してよい（原案どおり）。しかし総合スイートに穴が原因なら、テスト無しで直すのは invariant 違反になるため、工程3 `/spec` で回帰テストを先に作る。分析がこの2つを判定する。

## 帰結

- 工程7〜9 は `/integrate` の Phase A / Phase B になる。工程は消えず、コマンドが1本になる（ADR-0012 の `/spec` と同型）。
- 工程9b（総合テスト）と回帰ループが開発フローに追加される。playbook・計画書 §4 のフロー図に反映する。
- `/integrate` は GitHub MCP（PR マージ）＋ runner（全スイート起動）を内部で呼ぶ。Audit は関与しない（判定は工程8 で済んでいる）。
- 総合テスト CI（playbook ロードマップ Step3）は `/integrate` の総合テスト工程を自動化・前寄せする後継であり、二重ではなく移行関係にある。
