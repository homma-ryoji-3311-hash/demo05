# 単一アカウント運用では役割分離を機械強制できない。手続き的分離＋限界の明文化＋自己承認ガードで担保する

demo リポジトリ（`staff-report-system-demo`）の `/integrate 1` で、PR #1 を作成した GitHub アカウントと、工程8 の GO を出したアカウントと、`gh pr merge` を実行したアカウントが**すべて同一**になった。GitHub 自身は self-approve（`gh pr review --approve`）を「自分の PR は自分で承認できない」として拒否したが、これを GO **コメント**で迂回し、同じアカウントがマージまで完走した。結果、CLAUDE.md §7 が前提とする「PM の GO と 統合役のマージが別人格」という関所が形式的に崩れた（§6「勢いを承認と誤解する」の実例）。

なぜこれが起きるかは環境の制約であって手順の不注意ではない。単一 GitHub アカウントでは PR 作成者・ゲート判定者・マージ実行者を**identity の水準で分離できない**。GitHub のブランチ保護（作成者以外の review 必須）も、承認する第二の identity が存在しなければ機能しない。よって demo の間、機械強制ではなく**手続き的分離**でこれを担保し、その限界を明文化する。将来の多アカウント/CI 化まで「機械的に効くこと」の検証は保留する。

## 決めたこと

1. **demo（単一アカウント）では、役割分離は手続き的であって機械強制ではないと明記する。** 機械強制（作成者以外の review 必須・self-approve 不可）の検証は、多アカウントまたは CI（bot identity）を導入した環境でのみ成立する。demo の緑や「マージ完了」を、関所が効いた証拠として扱わない。

2. **agent は同一 run で自作 PR を GO・マージしない。** `/submit` で PR を作成したのと同じセッション/identity が、その PR に対して `/integrate` の Phase B（マージ）まで一気に走ることを禁じる。マージは統合役の帽子で**別ステップ**として、GO を独立に確認したうえで実行する。

3. **GO の正本は GitHub 上の判定コメント/approve のままとする（ADR-0014 を踏襲）。** ただし判定の author が PR の author と一致する場合、それは有効な GO として扱わない（下記ガード）。GitHub が self-approve を弾く挙動を、コメント GO でも再現する。

4. **`/integrate` に自己承認ガードを入れる。** Phase 判定とマージ直前で、GO/approve の author と PR の author を突き合わせ、**一致したら Phase B へ進まず停止**する。stderr 相当のメッセージで「単一アカウントのため機械分離が成立しない。別 identity の GO を得るか、手続き的分離（別ステップでの人手マージ）で進めること」を出す。

5. **限界の記録先を固定する。** 本 ADR に加え、単一アカウントで手続き的分離のまま進めた PR は `docs/metrics/incidents.md` に記録する（誰が author で、GO/マージを誰の帽子でやったか）。PM が事後に確認できるようにする。ADR-0007 の「代理判定は記録して PM が事後確認」と同じ立て付け。

## なぜ多アカウント化を今やらないか

bot/第二アカウントを統合役に割り当て、ブランチ保護で「作成者以外の review 必須」を有効化すれば identity 水準で分離でき、型を機械的に効くところまで検証できる。しかしアカウント・トークン・保護ルールの設定コストがかかり、いまの第一目的（再現可能な型の反復）に対しては過剰投資になる。よって demo 期間は手続き的分離で回し、限界を明文化して「機械強制は未検証」を自覚下に置く。多アカウント/CI 化は playbook ロードマップ（CI on main の Step）と同じタイミングで前寄せする余地を残す。

## なぜ手続き的分離でも型が壊れないか

型の不変条件は「不可逆操作は統合役の1点に集約」「判断は人・機械的手順だけをスキルが担う」（ADR-0014）である。identity が1つでも、(a) GO を独立ステップの成果物として GitHub に残し、(b) マージを GO 確認後の別ステップに切り、(c) 自己承認ガードで author==author を機械的に弾けば、**「自分の勢いで自分の PR を通す」経路だけは塞げる**。塞げないのは「同一人物が両方の帽子を被る」ことだが、それは demo の既知の限界として記録済みなので、型そのものの主張（機械分離は多アカウントで成立）は毀損しない。

## 帰結

- `/integrate` の SKILL.md に自己承認ガード（Phase 判定＋Phase B マージ前）を追記する。
- `docs/metrics/incidents.md` を新設し、PR #1 の自己マージを第1号として記録する（revert はしない。diff は fail-open 修正として 5 Whys・影響範囲・回帰テストが確認済みで、問題は「このコード」ではなく「前例」）。
- demo が bootstrap 前（`backend/`・`frontend/` 未生成）である間、`/integrate` Phase B の総合テストは空振りする。これは想定内で、インシデント記録に併記する。
- 多アカウント/CI での機械強制は本 ADR の未解決事項として残し、CI on main の導入時に再訪する。
