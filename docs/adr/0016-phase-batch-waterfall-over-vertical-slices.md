# 縦切り（tracer bullet）を廃し「全上流 → 全下流 → 全統合」のフェーズ型大バッチに切り替える。CI は下流フェーズ突入の必須前提に格上げする

このプロジェクトは当初、`to-issues` の縦切り分解（工程2）と目的C「Time-to-first-green」（playbook）に基づき、**1スライスを設計→仕様→テスト→実装→統合→緑まで端まで通す**縦切り（tracer bullet）で回す前提だった。狙いは、型（方法論）とハーネスの欠陥を**早期に1本で**炙り出すこと。

しかし運用上、per-slice の縦切りは1スライスごとに **PM → リーダー → 実装 → 統合**と役割の帽子を掛け替えることを強制する。ADR-0015（役割分離は手続き上のもの・シングルアカウント想定）の下では、この**帽子替えコストが毎スライス発生**し、文脈スイッチのスラッシングが停滞（作業の手待ち）を生む。PM の判断で、**縦切りを廃し、同種作業をまとめて処理するフェーズ型の大バッチ**に切り替える。掛け替え回数の最小化を、型の早期検証より優先する。

## 決めたこと

1. **全面大バッチ・縦切り廃止。** 開発は3フェーズの段階移行で進める。
   - **上流フェーズ**：全スライスの工程3〜5（`/spec` A/B・`/brief`・起票）を回し切る。
   - **下流フェーズ**：全スライスの工程6（`/slice`）を実装する。
   - **統合フェーズ**：全スライスの工程7〜9b（`/integrate`）を通す。
   フェーズは前段が**すべて**完了してから次段へ移る。playbook 目的C「Time-to-first-green」と「最初の縦切りは端まで通す」は撤回する。

2. **CI（playbook Step3：受け入れテスト CI ＋ マージキュー／required check）を「下流フェーズ突入の必須前提」に格上げする。** ADR-0010 の「初スプリントでは CI を組まない・最初の数スライスは統合役が全スイートをローカル実行で代替する」という**時期指定を上書きする**。
   理由：大バッチだと全 feature PR が統合フェーズ末尾に**一斉到着**する。ADR-0010 が必須化した「Require branches to be up to date before merging」により、**1本マージするたびに残り N-1 本が古くなり、各 PR は update → Phase A 再検証 → 工程9b 総合テストをやり直す**。統合役1人の手作業直列ではこの末尾クランチが破綻する。CI がマージキューで直列マージ＋再検証を自動化して初めて成立する。

3. **工程1 の末尾に「設計凍結ゲート」を置く。** 全 `source: PM`（answer-key に無い＝本当の設計）項目を解決・必要なら ADR 化してからでないと、大バッチ上流を開始しない。これは既存の心得「未定が残るまま `approved: true` にしない」（playbook ②）をバッチ全体へ拡張したもの。凍結後の変更は回帰スライス（fix-forward・append-only。ADR-0013・0014）で処理する。

## なぜ第一目的を書き換えてよいか（型の早期検証を手放す代償）

縦切りの価値は「型とハーネスの欠陥を早期に1本で炙り出す」ことにある。実際この repo は縦切り運用の中で `docs/memory-bank/hook-defect-slice-pattern-too-broad` と `hook-defect-unborn-branch` を**フローを回して**発見している。大バッチはこの早期露見を捨て、**最初の緑が統合フェーズまで出ない**——ハーネス欠陥は全上流を作り切った後に初めて露見し、仕様・テストの手戻りが最大化する。

この代償を受け入れる根拠は、**残るフィードバックの穴が限定的**なこと。工程4 が各受け入れテストを reference-mock に対して緑で検証するため、**`source: reference-mock` の項目は authoring 時点で自己検証済み**である。answer-key を持たないのは `source: PM` の項目だけであり、決定事項3の設計凍結ゲートがそこを先に潰す。残る「個別に緑な2 PR がマージ後に赤」（ADR-0010）と実装の合成バグは、決定事項2で前提化した CI／工程9b が捕まえる。

> **この段落の根拠は UI 層について成立しない（2026-07-15・ADR-0018）。** 現物の参照モックは文書ベースで
> **画面が無い**ため、工程4 の自己検証は **API 層にしか効かない**。UI の受入基準は全部 `source: PM`
> ＝ answer-key 無しであり、「穴は `source: PM` だけ・凍結ゲートが潰す」という限定が**画面については機能しない**。
> 大バッチ下では、**UI 仕様の誤りが統合フェーズまで一切露見しない**。実際、この穴から
> 「frontend を1行も書かずに緑」が下流を素通りした（`docs/metrics/incidents.md` #2）。
> ADR-0018 は工程4 に静的検知（UI spec の存在）、工程6 に実行時の反転と範囲 diff 判定を置いてこれを塞ぐ。
> **凍結ゲート（決定事項3）が画面設計を対象に含んでいるかは、上流フェーズ開始前に確認すること。**

## 帰結

- **原問題は本転換の副産物として解ける。** 「Phase A（工程7）完了後に統合役が次スライスへ移れず手待ちになる」停滞は、統合末尾の直列再検証を CI マージキューが自動化することで消える（決定事項2）。
- **CLAUDE.md §0（第一目的「再現可能な型の確立」）と playbook 目的C（Time-to-first-green）・工程2（縦切り）の書き換えが必要。** §0 は憲法級のため `docs/memory-bank/pending-*.md` に隔離し、**PM 承認後に本体へ統合**する（CLAUDE.md §10）。
- **playbook の全体像・開発フロー図をフェーズ型に改訂する。** per-slice の循環図から3フェーズの段階移行図へ。
- **ADR-0010 の CI 導入時期は本 ADR が supersede する**（スコープ＝統合役再検証の"当該スライスのみ"や必須付帯条件は維持。時期指定のみ上書き）。
- **作業者マニュアル（下流向け）をフェーズ運用へ改訂する。** 「1スライス完走」前提の記述をフェーズ単位に直す。
- **残リスク**：`source: PM` の誤りは下流フェーズで一斉露見しうる。回帰スライスで fix-forward するが、大バッチゆえ回帰が束で出る可能性がある。設計凍結ゲート（決定事項3）の厳格運用がこのリスクの唯一の抑え。
