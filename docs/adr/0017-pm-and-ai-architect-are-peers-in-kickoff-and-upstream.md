# キックオフと上流フェーズでは PM と AIアーキを承認まで含めて完全同列に扱う

単一アカウント運用（ADR-0015）では、PM も AIアーキも同一の人間が被る帽子である。にもかかわらず CLAUDE.md・playbook はキックオフ（Step 0・工程1・工程2）と上流フェーズ（工程3・4・5）で帽子を細かく割り当てていた——工程1 は「AIアーキ→PM 承認」、工程3/4 は「PM が叩く」、重量ゲートと `approved: true` は「PM」。同一人物なのに「いまはどの帽子か」を工程ごとに切り替える運用は、identity 水準の分離を1つも増やさないまま摩擦だけを生んでいた（`/spec` を AIアーキのセッションで進めたら手順違反、という無意味な停止が起きる）。

ADR-0012 は「author≠approver」を原則に掲げたが、その懸念（仕様の所有が静かに AIアーキへ移る）は **PM と AIアーキが別人格のときにだけ噛む**。単一オペレーターでは両者は最初から同一人物であり、identity 分離は ADR-0015 が「機械強制できない」と明記済みである。よってキックオフと上流に限り、帽子の別を撤廃して**完全同列**にする。

## 決めたこと

1. **キックオフ（Step 0・工程1・工程2）と上流フェーズ（工程3・4・5）では、PM と AIアーキは完全同列とする。** どちらの帽子でも各工程の作業（grill への応答・基本設計・issue 分解・仕様表作成・翻訳・指示書起票）を実行でき、相互に代行してよい。「この工程は PM の帽子でなければ叩けない」という手続き上の制約を、この範囲に限って撤廃する。

2. **承認ゲートも同列に含める。** 工程1 の設計凍結 GO、工程3/4 の `approved: true`、上流の重量ゲート（G1）は、**PM／AIアーキのどちらが押してもよい**。ただし `approved: true` は口頭でなく git に残す（成果物として残す運用は ADR-0012 のまま不変）。

3. **適用範囲はキックオフ＋上流に限る。下流（工程6）と統合（工程7〜9b）は不変。** 工程8 の層境ゲート GO/NO-GO は PM（代理はリーダー1名、ADR-0007）、不可逆操作（`main` 前進・`gh pr merge`・DB マイグレーション）は統合役ただ1人（ADR-0014）。ADR-0015 の自己承認ガード（PR author == GO author を弾く）は統合フェーズでそのまま維持する。

4. **承認ゲートは identity 分離ではなく「手続き的な停止点」として残す。** 同列化しても、Phase A と Phase B の間で停止し `approved: true` が git に無ければ `acceptance/` へ進めない hook（`protect-paths.sh`、ADR-0012）は撤廃しない。押せる人を広げるだけで、生成と評価を別ステップに割る機構そのものは保持する。

## なぜ ADR-0012 と矛盾しないか

ADR-0012 が守りたかったのは「AI が自問自答で仕様を確定させない」ことであり、その担保は (a) grill の答えを人間から取る、(b) 各受入基準に `source:` を記録する、(c) Phase A/B を停止で割る、の3点だった。本 ADR は**「誰が押せるか」を広げるだけ**で、この3機構には一切触れない。単一オペレーターでは PM も AIアーキも同一人物なので、"author≠approver" は元々 identity では成立していない（ADR-0015 の明文化どおり）。よって自己証明への防波堤（`source:` 記録・Phase 間の停止点）は無傷であり、ADR-0012 の実質は保たれる。

## なぜキックオフ＋上流に限るか

型の不変条件は「不可逆操作は統合役の1点に集約」（ADR-0014）である。上流の成果物（`docs/spec/`・`acceptance/`・`golden/`・指示書）はすべて `spec/*` ブランチ止まりで `main` を進めない。したがってここを同列化しても不可逆性の集約は崩れない。一方、統合フェーズを同列化すると「自分の勢いで自分の PR を通す」経路が開き、ADR-0015 が自己承認ガードで塞いだものを再び開けてしまう。よって同列化は「main を進めない工程」だけに限定する。

## 帰結

- CLAUDE.md §0 の ADR リストに 0017 を追加。§4 の `/spec`・`/brief` の主体を「PM／AIアーキ」に。§7 に「キックオフ＋上流は PM／AIアーキ同列（承認含む）」の一文を足す。
- CONTEXT.md の基本設計・仕様表・翻訳の所有表記を、キックオフ＋上流の同列扱いに整合させる。
- `docs/playbook.md` のフロー図（工程1〜5・G1・approved ゲート）と早見表の担当列を「PM／AIアーキ」に更新する。
- 下流・統合の担当（工程6〜9b）、ADR-0015 の自己承認ガード、`protect-paths.sh` の停止機構は無変更。
