# 受け入れテストは API 層と UI 層の二層を必ず持つ。golden 撮影の可否と UI 検証の要否は別の判断とする

下流フェーズで、指示書「3. 触ってよいファイル範囲」に `frontend/app/**` が明記されているスライスが、**frontend を1行も書かないまま「受け入れテストが緑」で完了報告された**。テストは `@playwright/test` の `request` フィクスチャだけで書かれており、`page.goto()` を1回も呼んでいなかった。つまり**スイートは最初から画面を検証していない**。実装メンバーは憲法 §2 の完了の定義（3つとも機械判定）を字義どおり満たしており、規律違反はしていない。

原因は工程4（`/spec` Phase B・翻訳）にある。ADR-0008 は「参照モックの画面を golden として撮る」と定め、ADR-0005 は「同じスイートが参照モックに対して緑・実装前の backend に対して赤——この反転が翻訳の正しさの機械的な証明」と定めているが、**どちらの前提（参照モックに画面がある）も実際には崩れていた**——参照モックは文書ベースで、撮影する画面が無い。ADR-0008 はこの前提が崩れたときの縮退経路を書いていない。結果、翻訳の担当（PM／AIアーキ）が現場で「**pixel 比較ができない**」を「**UI 検証をしなくてよい**」へ独断ですり替え、API-only のスイートを書いた。前提の崩れは ADR を書き換える事由（＝上流へ差し戻して PM の判断を仰ぐ事由）だったが、工程4 の止まる条件に該当項目が無いので、誰にも報告されずに通過した。

**これは個人のさぼりではなくハーネスの欠陥である。** そう位置づけないと対策が「気をつける」という宣言に落ち、憲法 §10 の強制力の階段（宣言 → 実行時強制 → 事後検証）を逆走する。

## 決めたこと

> **本命は決定6（範囲＝予告）。** 決定1〜5 はすべて「上流が正しく翻訳する」に依存しており、
> **今回失敗したのがまさにその経路**である。決定6 だけが、上流の翻訳が間違っていても下流で落ちる。
> 時間が無ければ決定6 だけ読んで、実装も決定6 から着手してよい。

1. **`acceptance/` は二層構成を必須にする。** 1スライスにつき最低2本：

   | ファイル | フィクスチャ | 検証するもの |
   |---|---|---|
   | `acceptance/<feature>/<name>.api.spec.ts` | `request` | HTTP の契約（ステータス・ボディ・エラー） |
   | `acceptance/<feature>/<name>.ui.spec.ts` | `page` | ユーザーに見える振る舞い（`page.goto()` 必須） |

   **UI spec が 0 本のスライスは工程4 を完了できない。** 画面を持たないスライス（バッチ等）は、仕様表の画面要件に「画面なし」＋理由を明記した場合にかぎり免除する。免除は仕様表の `approved: true` に含まれる＝**PM／AIアーキが明示的に押した**ことになる。黙って省略できる経路を閉じるのが目的。

2. **golden 撮影の可否と UI 検証の要否を分離する。** 撮れるなら ADR-0008 のとおり `toHaveScreenshot()` で pixel 比較する。撮れないなら **role/label ベースの DOM アサーション**（`getByRole` / `getByLabel` / `getByText`）へ**縮退する**。ADR-0008 が代替案として却下した「DOM の構造アサーションのみ」は、**golden が撮れる状況での比較検討**であって、撮れない状況での代替を否定したものではない。撮れないときに何もしないのは、そのどちらでもない第三の選択肢であり、これを禁じる。

   縮退したスライスでは「参照モックと画面が一致」は保証されない。**それは受け入れる。** 撮れない以上どのみち保証できず、保証できないことと検証しないことは別だからである。

3. **参照モックに画面が無いなら、画面の受入基準は `source: PM` になる。** answer key に無い＝本当の設計（仕様表テンプレ）。工程4 の翻訳者が現場で埋めてよい空白ではなく、**工程3 で PM が決め切る**。

   **この帰結は重い**（下の「認めること」を参照）。画面設計そのものが `docs/design/overview.md` に無いなら、それは**工程1 の設計凍結ゲート（ADR-0016 決定3）の漏れ**であり、上流フェーズを止めて工程1 へ戻る事由になる。凍結ゲートの対象は overview.md の `source: PM`、工程3 で決めるのは各スライスの UI 受入基準——**別の成果物**なので混同しない。

4. **UI spec には answer key が無い。工程4 では緑にならない。それを正常とし、静的に検知する。**

   参照モックに画面が無い以上、`ui.spec.ts` を**緑にする方法が工程4 には存在しない**（frontend は下流フェーズで実装される）。したがって ADR-0005 の「参照モックで緑＝翻訳が正しい証明」という反転は、**構造的に API 層にしか効かない**。工程4 の順序は：

   ```
   3.  翻訳：api.spec.ts ＋ ui.spec.ts（page.goto 必須）
   4.  参照モック起動 → api.spec.ts が【緑】            ← API 翻訳が正しい証明（ADR-0005）
       ui.spec.ts は【赤のまま】。これは正常            ← 緑にする手段が上流に無い
   4b. 静的検知：画面ありスライスに *.ui.spec.ts が存在し page.goto( を含むか
                                                       ← UI 翻訳に効く唯一の機械判定
   5.  golden（撮れるスライスのみ・実装より先に）
   6.  backend だけ起動 → api.spec.ts も【赤】          ← 下流に渡せる証明
   7.  PR
   ```

   **ステップ 4b は実行ではなく静的検知である**（`grep`／CI）。frontend が存在しなくても判定でき、そこが要点。API-only のスイートを書くと `*.ui.spec.ts` が0本になり、**そこで落ちる**。

   **UI spec が「正しい」ことは工程4 では証明できない。** 存在すること・画面を叩いていることまでしか機械で言えず、内容の正しさは **PM の重量ゲート**（`acceptance/` に触るので常に重量・ADR-0007）が読むしかない。ADR-0008 が golden で追い出したはずの人の目が、参照モックに画面が無い以上、上流に戻ってくる。

5. **frontend を落とす反転確認は工程6（`/verify`）で行う。** frontend が実在する下流フェーズで初めて実行できる：

   ```
   ui.spec.ts が緑 → harness_stop(frontend) → 再実行 → ui.spec.ts が【赤】に転ぶ
   ```

   転ばなければ、その緑は frontend 由来ではない。**上流の翻訳欠陥を、下流が実行時に捕まえる最後の網**。4b（静的）と重複するが、静的検知は `page.goto()` の有無しか見ないので、両方要る。

6. **「触ってよいファイル範囲」は許可であると同時に予告である。** 指示書 §3 に挙げたディレクトリに **diff が1行も無いまま完了報告してはならない**。`git diff --name-only` と §3 の突き合わせは機械判定できる。未着手のディレクトリがあれば、**テストが緑でも停止して報告する**（工程6-4 `/verify` 判定3）。

   **これが対策の本命。** 決定1〜5 はすべて「上流が正しく翻訳する」に依存しており、今回失敗したのはまさにその経路である。決定6 だけが**上流の翻訳が間違っていても下流で落ちる**。上流への信頼を二重化しない。

## 認めること（この ADR が解決しないもの）

**参照モックに画面が無い＝画面の answer key がそもそも存在しない。** これはテストの書き方の問題ではなく、**このプロジェクトの前提の問題**である。憲法 §0 は「Repo Base の挙動を再実装する」と書くが、画面については再現すべき挙動が渡されていない。UI 仕様は全部 `source: PM` の新規設計になる。

本 ADR はこの穴を塞がない。塞ぐのは工程1（設計凍結ゲート）であり、**上流フェーズを始める前に PM が画面設計を決め切っているか**を確認する必要がある。決まっていないなら、UI spec は「PM が決めていない仕様」を検証することになり、二層にした意味が消える。

## 帰結（導入時に必ず要るもの）

**この2つは ADR-0018 の前提であり、片方でも欠けると工程4／工程6 が物理的に走らない。**

- **ADR-0005 の CI ジョブのスコープを API 層へ縮める**（ADR-0005・playbook 工程4 の文言は**改訂済み**。残るのは CI の実装）。ADR-0005 の初稿は「`spec/*` PR では参照モックに対して**スイートが緑**」を CI に要求していた。本 ADR は `ui.spec.ts` が工程4 で赤いのを正常と定めたので、**そのまま実装すると全 `spec/*` PR が落ちる**。判定対象は **`*.api.spec.ts` のみ**。`ui.spec.ts` を CI が見ないという意味ではなく、**参照モックに対しては見ない**という意味である（見る場所は工程6 の `/verify`）。
- **接続先の環境変数を2本にする。** 現状 `ACCEPTANCE_BASE_URL` は1本（憲法 §5・**改訂は PM 承認待ち**）だが、工程6 では `api.spec.ts` が backend :3000、`ui.spec.ts` が frontend :3001 を**同時に**叩く。1本では二層スイートを1回で走らせられない。`ACCEPTANCE_BASE_URL`（API）＋ **`ACCEPTANCE_UI_BASE_URL`**（画面）へ分ける。
  - 副作用：工程4 の止まる条件「参照モックで `ui.spec.ts` が緑になってしまう」は、UI 側の向き先を :8000 にしたときだけ意味を持つ。**工程4 では `ACCEPTANCE_UI_BASE_URL` を参照モックへ向ける**（緑になったら参照モックに画面があった＝ADR-0008 の前提が生きている＝止まって PM へ報告）。**2本化が入るまで、この止まる条件は発火しない。**
- **工程4 の翻訳量が増える**（UI spec の分）。実行の往復は増えない（4b は静的検知）。
- **`ui.spec.ts` は工程4 から下流フェーズ完了まで赤いまま main に居座る。** 「テストが赤いのは正常」の期間が API 層より長い。**赤の放置に慣れる**のが最大の副作用で、だからこそ 4b と判定3 を機械判定へ落とす必要がある。
- `ui.spec.ts` は frontend と backend の**両方**が起動していないと緑にならない。`/implement` は runner で2プロセスを立てる（`harness_start` は 1 app_dir = 1 プロセス）。
- **`frontend/test-harness.runtime.json` の `NEXT_PUBLIC_API_BASE` は backend 固定**で、runtime.json の `env` は親環境を上書きする（`process_manager.py`）。`harness_start` に env 上書き引数は無い。つまり **frontend を参照モックへ向けることは現状できない**。決定4 が実行ではなく静的検知である理由の一つでもある。
- 画面（Next.js）は Java/Spring 移行の対象外なので、`ui.spec.ts` も golden と同じく**バックエンドを差し替えても不変の資産**になる（ADR-0008 と同じ理屈）。

## 考慮した代替案

- **完了の定義に4つ目「画面が動く」を足す。** 憲法 §2 の「すべて機械判定」を壊し、初級者に主観判定を戻す（ADR-0008 が消したはずのもの）。却下。
- **指示書「4. 貼り付け用の枠」に「frontend も実装すること」と書く。** 宣言止まり。テストが検証しない範囲は必ず省略される——今回それが実証された。憲法 §10「破られ続けるルールは hook に昇格」の逆走。却下。
- **工程9b の総合テストでまとめて検知する。** フィードバックが右に寄り、全スライス実装後に一斉に赤が出る。ADR-0008 の「フィードバックは左に寄せる」に反する。却下。
- **参照モックに画面を足して golden を撮れるようにする。** 参照モックは answer key であり read-only（ADR-0005）。こちらが画面を足せば、**答えを自分で書いてから答え合わせをする**ことになる。却下。ただし「参照モックが本当に撮影不可か」の再調査は `/flywheel` の観察項目として別に立てる。
- **工程4 で frontend を起動し、落として反転を見る**（本 ADR の初稿はこれを中核に据えていた）。**成立しない。** 工程4 は上流フェーズで frontend は未実装だから、上げても落としても UI spec は赤のままで、反転が情報を持たない。加えて frontend の向き先は runtime.json で backend 固定であり、`spec/*` からは編集できない（憲法 §1-4）。**実行時の反転確認は frontend が実在する工程6 でしか意味を持たない**（決定5）。

## 未解決（＝この ADR は宣言止まりの部分を残している）

強制力の階段（宣言 → 実行時強制 → 事後検証）を飛び級しない方針だが、**本 ADR の決定は現状すべて宣言である。skill・hook・CI には1行も入っていない。**

**作業表の正本は `docs/memory-bank/pending-fullstack-completion.md`**（項番・担当・優先順位はそちら。ここに二重管理しない）。要点だけ：

- **前提2件**（CI スコープ縮小・base URL 2本化）が入るまで、**本 ADR は物理的に動かない**——工程4 の PR が落ち、工程6 の二層スイートが1回で走らない。
- **決定6（判定3）と決定4（4b）が入るまで、今回の事故を再発させる。**
- 決定1〜5 を実行する主体は `/spec`・`/verify`・`/implement` の各 skill であり、**そこが更新されない限り、この ADR はどの経路からも読まれない**。今回の事故と同じ構図（`/verify` の skill 本文は ADR-0008 制定後も目視比較のまま残っていた）を繰り返さないこと。

もう1点。**画面設計が `docs/design/overview.md` に無いなら、本 ADR は空回りする**（「認めること」参照）。UI spec が「PM が決めていない仕様」を検証することになる。上流フェーズ突入前に確認すること。これも作業表に載せてある。
