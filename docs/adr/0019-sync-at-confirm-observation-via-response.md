# 確定に同期で相乗りするサーバ内処理の観測は、確定レスポンスの拡張に統一する

Phase 3 以降、「独立した画面もエンドポイントも持たず、**報告確定（`POST /reports/:id/confirm`）の同一処理内で同期実行される**サーバ内処理」を扱うスライスが連続する。slice-11（案件キー紐づけ・INCIDENTS）、slice-12（突合・MASTER_SUMMARIES）がその典型で、以降 slice-23（AI 追加質問）等も同型になる。

これらは**参照モックが HTTP のワイヤ契約を持たない**（reference-mock は振る舞い＋データモデルの文書で、リクエスト/レスポンス形を規定しない）。そのため工程3（`/spec` Phase A）で毎回、grill の最初の質問が「**その処理結果をどう HTTP で観測するか**（確定レスポンスに含める / 専用 GET / 別 feature 経由）」になり、同じ判断を繰り返していた（slice-11・slice-12 で 2 回連続）。同じ設計判断を繰り返すのは、上流の grill 枠を食い、スライス間で観測経路がばらつく温床になる。

**これは個人の判断のブレではなく、方針の未確定（＝ハーネスの欠陥）である。** 1 本の決定に畳んで既定化すれば、次回以降の grill から「観測経路」の質問を落とせる。

## 決めたこと

1. **確定同期系スライスの観測経路は「確定レスポンスの拡張」に統一する。**
   確定操作の同一処理内で同期実行される内部処理（＝AC-5 型「確定のたび同期実行」）の結果は、
   `POST /reports/:id/confirm` の**レスポンスに新しいトップレベルキーを1つ足して**返す。
   例: slice-11 は `projects` / `incidents`、slice-12 は `master_summaries`。
   - 専用 GET エンドポイントは足さない（確定と観測を2ステップに割らない）。
   - 別 feature（例 slice-08 のスキルシート）経由の間接観測にしない（結合度を上げない）。

2. **既存キーは壊さない・追加のみ（後方互換）。**
   レスポンス拡張は既存キー（`status` / `confirmed_summary` 等）を不変に保ち、キーの**追加のみ**とする。
   これにより slice-03 以降の既存 acceptance が回帰しない（実測: slice-11 で確認済み）。

3. **観測対象の永続レイヤーは feature 境界を越えて直接触らない。**
   確定同期処理は自 feature の repository（例 projects / master-summaries）にのみ書き、
   他 feature の read 先（例 slice-08 の masters）を直接更新しない。跨ぐ配線が要るときは
   `no-cross-feature-import` に従いポート＋合成ルート注入（ADR-0011・home の read ポート前例）。

4. **`/spec` の grill から「観測経路」の質問を落とす。**
   工程3 の grill は、確定同期系では**Q1（観測経路）を確定済み前提**とし、
   **レスポンスに足すキーの形（キー名・値の shape）だけ**を PM に pin する。
   非同期化（夜間バッチ等）に切り替える判断が出たときは、そのスライスの仕様表に
   `source: PM` で明記し、本 ADR の例外として扱う（既定は同期＋レスポンス拡張）。

## 認めること（この ADR が解決しないもの）

- **これは観測（テスト可能性）の統一であって、実行タイミングの決定ではない。** 同期/非同期の選択は
  各スライスの AC（例 slice-12 AC-5）と PM 決定（overview §8.5）が持つ。本 ADR は「同期でやると決めたものの**観測をどう HTTP に出すか**」だけを既定化する。
- **確定レスポンスが太る。** 確定同期系が増えるほどレスポンスにキーが積み上がる。1 スライス=1 トップレベルキーの規律で肥大を抑えるが、将来的に「確定サマリ専用の集約レスポンス」へ再編する余地は残す（後続の ADR で扱う）。
- **参照モックの HTTP 契約不在そのものは直さない。** 実行オラクル（`tools/reference-mock-server/`）を工程4 で拡張する運用は変えない。本 ADR は「拡張時の観測経路を毎回決め直さない」ための既定である。
