## 1. プロダクトの目的

業務報告システム。3 つの中核フローを持つ（`README.md` より）。

1. **業務報告**: 入力 → AI 整形 → 本人確認・修正 → 確定保存
2. **スキルシート生成**: 蓄積された業務報告からスキルシートを生成・確定・配布
3. **推奨案件**: 対向システムから取得した案件を営業担当が開示判定 → エンジニアに提示 → エンジニアが意思表示

## 2. アクターと権限（`CONTEXT.md` 準拠）

| アクター | 主な目的 | 権限範囲 |
|---|---|---|
| エンジニア | 業務報告の入力、AI整形結果の確認・確定、自身のスキルシート閲覧、推奨案件への意思表示 | 自分のデータのみ |
| 営業担当 | 担当エンジニアの状況確認、スキルシート生成・確定、推奨案件の開示判定 | 担当グループのエンジニアのみ |

権限境界は画面表示だけでなくバックエンドで必ず強制する（`必要画面・機能一覧.md` §1、全画面仕様の「権限・表示制御」で共通）。

> **HR / 担当者ロールは本設計に含めない**（§0 の決定）。雑感（メンタル面）の扱いは
> 「本人と閲覧限定担当者のみ」の制約だけを画面仕様から引き継ぎ（05 画面参照）、
> 専用ロール・専用画面は将来スコープとする。

## 4. データモデル概要（各画面仕様「関連データ」より集約）

正式なスキーマ設計はスライス実装時に確定する。ここでは画面仕様が要求するエンティティの粒度のみ列挙する。

| エンティティ（画面仕様表記） | 概要 | 主に参照する画面 |
|---|---|---|
| `USERS` | エンジニア／営業担当。ロール・所属グループを持つ | 全画面 |
| `REPORTS` | 業務報告。下書き／確定、raw_text、AI整形後本文、確定要約（対応案件／インシデント／成果／課題／抽出スキル） | 2, 3, 5 |
| `QUESTION_SETS` | グループ別・バージョン付きの設問セット。報告作成時点の版を保持 | 3 |
| `REPORT_PROJECTS` / `PROJECTS` | 報告に紐づく対応案件 | 5 |
| `INCIDENTS` | インシデント状態（最新状態へ更新、単純追記ではない） | 5 |
| `GENERATED_SHEETS` | 生成済みスキルシート。ファイル名・生成日・署名付きURL・履歴（世代） | 2, 6, 7, 8 |
| `TEMPLATES` | グループ別 Excel テンプレート・アンカーマップ・有効版 | 8, 10 |
| `MASTER_SUMMARIES` / `SKILLS` | 突合済みマスター元データ・スキル集計。スキルシート生成の入力 | 8 |
| 対向システム取得案件・承認状態・応答ログ | 推奨案件フロー（提案 → 営業承認 → エンジニア応答） | 6, 9 |

**生報告ログとマスターデータの分離**（`必要画面・機能一覧.md` §3「報告確定・突合」）: 確定した `REPORTS` は不変ログとして保存し、
`MASTER_SUMMARIES` 側は確定のたびに案件キー単位で増分マージする別データとして扱う。この分離は本設計でも維持する。

## 5. AI 利用の原則（`README.md`・全画面仕様共通）

- AI 呼び出しは提供元非依存の `Summarizer` 抽象化層の背後で行う（プロバイダー差し替え可能。ADR-0003）。
- AI は本文にない数値・事実を創作しない。確定前に必ず人が確認・編集できる。
- 不足・不確実な箇所は要確認フラグとして表示する（AI整形結果確認画面＝画面4。ただし §6 のとおり answer key 欠落）。
- スキルシート生成の AI は言い換えのみを担当し、出力はアンカー対応 JSON スキーマに固定する（画面8「生成の3フェーズ」）。
- 推奨案件の生成・評価・順位付けに本システムの AI は関与しない（対向システム由来。画面9）。
