# ADR 草案 0019: 工程4 は「テストが AC の実質を固定していること」を機械で証明する

> **AIアーキ（Harness-Keeper）確定待ち。** ADR は `/flywheel` の書き戻し先として「Harness-Keeper 単独で確定してよい」が、
> `docs/adr/` は permission deny のため草案を memory-bank に隔離する（`pending-adr-0011-*` と同じ慣習）。
> 起票日: 2026-07-17 / 起案: 下流セッション（/flywheel）
> 関連: ADR-0001（ブラックボックス）/ ADR-0012（`/spec` 2フェーズ）/ ADR-0018（UI 層の検証）

## 文脈

工程4（`/spec` Phase B・仕様表 → `acceptance/` への翻訳）の産物に、**7スライス中6スライスで欠陥**が出た。無傷は slice-01 のみ。

| issue | スライス | 内容 |
|---|---|---|
| #15 | slice-02 | AC の実質を固定していない（要約しないスタブが 4/4 通る） |
| #19 | slice-03 | `confirm.ui.spec` が AC-1 の Given を落とす |
| #20 | slice-04 | `list.ui.spec` が strict mode と UI-AC で矛盾し、正しい実装が赤になる |
| #21 | slice-05 | `previous.ui.spec` が「前回なし」の Given を落とす ＋ 待機条件欠落で偽の緑を出す |
| #25 | slice-06 | 認可の検証が1エンドポイントずつ／`role`・upsert が未検証 |
| #26 | slice-07 | `home.ui.spec` が strict mode で矛盾／AC の「今日」がオラクルに無い |

全て PR #1 で一括作成された同一バッチの産物である。個別の不注意ではなく、**手順が構造的に検知できない欠陥**と見るべき。

## 現行の機械判定が測っているもの

`/spec` Phase B には既に3つの機械判定がある。

- **手順4**: 参照モックで `api.spec.ts` が緑 ← API 翻訳が正しい証明
- **手順4b**: 静的検知。画面ありスライスに `*.ui.spec.ts` が存在し `page.goto(` を含むか
- **手順6**: backend で `api.spec.ts` が赤 ← 下流に渡せる証明

そしてスキル本文はこう締めている:

> **UI spec の「正しさ」は工程4 では証明できない。** …存在すること・画面を叩いていること（4b）までしか機械で言えず、内容の正しさは **PM の重量ゲート**が読む。

**この宣言どおりに 6回すり抜けた。** PM の重量ゲートは、6回とも欠陥を止められなかった。宣言は機能していない。

## 決定の核心: 反転テストは「触っているか」しか証明しない

手順4・6 の反転（モックで緑／backend で赤）は強力だが、**証明しているのは「テストが当該エンドポイントを叩いていること」だけ**である。

backend にエンドポイントが無ければ 404 が返る。**assert がどれだけ弱くても赤になる。** したがって反転は、実質を固定していないテスト（#15・#25）を素通しする。手順6 の「緑になったら停止。実装済みか、テストが何も検証していないかのどちらか」はこの穴を認識しているが、**赤であることは何も保証しない**。

欠陥は2クラスに分かれ、必要な検査が異なる。

| クラス | 症状 | 該当 | 反転で捕まるか |
|---|---|---|---|
| **1. 実質未固定** | AC を満たさないスタブが緑になる | #15・#25 | **捕まらない**（404 で赤くなるため） |
| **2. 矛盾ペア** | 正しい実装が原理的に赤になる | #19・#20・#21・#26 | 捕まらない（実装前は両方赤） |

## 決定

`/spec` Phase B に、2クラスそれぞれに対応する機械判定を足す。**強制力の階段を「宣言（PM が読む）」から「静的検知・実行時強制」へ1段上げる**（ADR-0012 の2フェーズ構造は変えない）。

### 手順 4c（新設・静的検知）: 矛盾ペア検査 — クラス2 用

> **同一の `*.ui.spec.ts` 内で、2本以上の test が同じ URL を `page.goto()` し、かつ状態を変える操作（`test.use({ storageState })` の上書き、または arrange の API 呼び出し）を持たないなら、停止して PM へ返す。**

**根拠**: 2本が同じ URL を同じ状態で叩くなら、両者は**同じ画面**を見ている。異なる状態の AC を2本に分けたなら、状態を作る arrange が必ず要る。無いなら Given が落ちている。

**既知データでの的中率は 4/4、誤検出 0/1**:

| ui.spec | 2本の goto 先 | `test.use` | arrange | 判定 |
|---|---|---|---|---|
| `confirm.ui.spec`（#19） | 両方 `/reports/new/review` | なし | なし | **検出** |
| `list.ui.spec`（#20） | 両方 `/reports` | なし | なし | **検出** |
| `previous.ui.spec`（#21） | 両方 `/reports/new` | なし | なし | **検出** |
| `home.ui.spec`（#26） | 両方 `/home` | なし | なし | **検出** |
| `auth.ui.spec`（健全） | `/login` と `/reports/new` | **あり** | — | 通過（正しい） |

`auth.ui.spec` が唯一健全なのは、`test.use({ storageState: { cookies: [], origins: [] } })` で状態を明示的に上書きし、かつ2本が別 URL を叩いているため。**健全な spec だけが持つ特徴を、そのまま判定条件にできる。**

4b と同じく grep で判定でき、**frontend が存在しなくても効く**。

### 手順 6b（新設・実行時強制）: スタブ反転 — クラス1 用

> **AC を満たさない最小スタブに対して `api.spec.ts` を走らせ、赤になることを確認する。緑なら翻訳が実質を固定していない＝停止して PM へ返す。**

**根拠**: #15 は「要約しないスタブが 4/4 通る」という**想像**で発見された。想像でできることは機械にもできる。スタブを実際に書いて走らせれば、実質未固定は確実に落ちる。

**コスト**: スライスごとに「振る舞いの中身を持たないスタブ」を書く必要がある。安くはない。ただし参照モックが answer key として既にあるため、**その振る舞い部分だけを抜いたもの**を作ればよく、ゼロからの実装ではない。

このコストが許容できないなら、代替として **PM の重量ゲートに「このテストを通す最小のインチキ実装を口頭で言えるか？」という問いを明示的に課す**（＝宣言に留める）。ただしその場合、本 ADR は「クラス1 は宣言のまま」と正直に記録すること。**6回すり抜けた宣言に再び頼る選択である**と認識したうえで決めるべき。

## 却下した案

- **4観点のチェックリストを `/spec` 本文に足すだけ（宣言）**: 現行が既に「PM の重量ゲートが読む」という宣言であり、6回破られている。同じ段に留まる提案は、`/flywheel` の「書いてあるのに破られた → 昇格させる」に反する。
- **下流の `/verify` で検知する**: 遅すぎる。下流が実装を終えてから「テストが間違っていた」と分かるのが現状であり、6回とも下流の時間を浪費している。工程4 で止めるのが本 ADR の目的。

## 帰結

- `/spec` Phase B の手順が 4c・6b の分だけ重くなる。上流の1回のコストで、下流の6回の手戻りを防ぐ。
- 4c は既存 issue（#19・#20・#21・#26）の修正時にも使える。**修正後の spec が 4c を通ることが、修正の機械的な証明になる。**
- 4c は「同じ URL を叩く2本」を機械的に疑うため、正当な理由でそうする spec には `test.use` か arrange を書く義務が生じる。**これは望ましい副作用**である（Given が明示される）。

## 未解決

**クラス2 の亜種「待機条件欠落による偽の緑」（#21）は 4c では捕まらない。** `toBeVisible()` が fetch 解決前の初期レンダリングを拾い、誤った実装に緑を与える問題。落ち着いた後の DOM では偽であるにもかかわらず通り、読み込み状態を持つ**正しい実装のほうが赤になる**という逆転が起きる。

これは実装が存在して初めて現れるため、工程4 では原理的に検知できない可能性がある。下流 `/verify` での検知を別途検討する（実 DOM と落ち着いた後の状態の突き合わせ）。
