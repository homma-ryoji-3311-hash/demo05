# CLAUDE.md 昇格候補: Stop hook が憲法 §2 と矛盾している（緑を報酬にする）

> **PM 承認待ち。** 憲法（CLAUDE.md）§6 への追記提案 ＋ AIアーキへの hook 修正提案。
> 起票日: 2026-07-17 / 起案: 下流セッション（/flywheel）

## 観察

slice-05 の実装中、Stop hook（`.claude/hooks/stop-gate.sh`）が **3ターン連続でブロック**した。

```
STOP GATE: 受け入れテストが緑ではありません（status=failed）。
緑になるまで /implement のループを続けてください。
```

3回目は**実装を1行もしていない会話ターン**（PM の作業順を答えただけ）でも発火した。hook は直前のテスト status しか見ておらず、そのターンが実装だったかを見ていない。

## なぜこれが重要か（本質）

このセッションで、**受け入れテスト 4/4 が緑になった状態から、意図的に赤へ戻した**。

経緯:
1. slice-05 の UI を素直に実装したら 4/4 緑になった
2. 緑を疑ってアクセスログと実 DOM を確認したところ、**落ち着いた後の画面に当該テキストは存在しなかった**
3. fetch 解決前の一瞬のレンダリングを `toBeVisible()` が拾っていただけだった（＝偽の緑）
4. 読み込み状態を持たせる正しい実装に直した結果、**2本目は赤に戻った**

つまり **Stop hook を満たす唯一の道は、偽の緑に戻すこと**だった。hook は「一瞬だけ事実と異なる表示を出す実装」に報酬を与え、正しい実装を罰する。

**これは憲法 §2 の「緑 ≠ 仕様充足。最終判定は層境ゲート（PM の GO/NO-GO）が行う」と正面から矛盾する。** 憲法は緑を十分条件ではないと宣言しているのに、hook は緑を停止の必要条件にしている。

## 構造的な問題

hook が区別できないものが2つある:

1. **上流欠陥由来の赤**（下流では原理的に解消できない）と、実装不足の赤
2. **偽の緑**（テストが通るが仕様を満たさない）と、正しい緑

1 について: slice-05 の ui.spec 2本目は、`playwright.config.ts` が UI を staff01 に固定している一方 staff01 は前回確定報告を持つため、「前回が無い」状態が成立しない（#21）。`acceptance/` は下流から変更禁止（憲法 §1-3）なので、下流にできることは何もない。それでも hook は「緑になるまでループを続けろ」と言う。

hook 自身が例外を列挙している（「5ファイル以上変更した → 影響範囲を報告」等）ため押し切らずに済んだが、**例外に当たらなければ、モデルは緑にするために実装を歪める圧力を受け続ける**。

## 提案

### 1. hook の修正（AIアーキ確定・`.claude/hooks/stop-gate.sh`）

- **実装していないターンでは発火しない。** 直前のターンで `Edit` / `Write` が使われていないなら、テスト status は判定材料にならない。
- **赤の帰責が上流にあると報告済みなら、例外に加える。** 現行の例外リスト（同一エラー2回・3回リトライ・5ファイル以上）に「上流 issue に差し戻し済み」を足す。判定は「直近の報告に issue 番号への言及があるか」で機械的に取れる。

### 2. CLAUDE.md §6 への追記（PM 承認必須）

§6「非自明な落とし穴」に既にある行:

```
- **Stop hook は8連続ブロックで override される。** 完了の最終担保は統合役の runner 再検証であり、hook ではない。
```

これに1文足す:

```diff
- **Stop hook は8連続ブロックで override される。** 完了の最終担保は統合役の runner 再検証であり、hook ではない。
+ **hook は緑しか見ない。緑にするために実装を歪めそうになったら、それが上流へ返す合図。**
```

理由: 既存の行は「hook は最終担保ではない」と言っているが、**hook に逆らってよい状況の判断基準**を与えていない。実際このセッションでは、hook を満たす道が偽の緑への回帰しかないと気づくまでに、緑を捨てるという不自然な判断が必要だった。基準を1行で持たせる。

## 強制力の階段

- 現在: **実行時強制**（hook が存在し、exit 2 でブロックしている）
- 提案: **階段は上げない。** 既に実行時強制の段にあり、問題は段ではなく**判定条件の精度**。飛び級も昇格もせず、同じ段のまま条件を直す。

§6 への1行追記は宣言だが、これは hook の昇格ではなく「hook に逆らう判断基準」を人間の言葉で置くもの。強制力を足すのではなく、**強制力の誤作動時の逃げ道を明示する**。
