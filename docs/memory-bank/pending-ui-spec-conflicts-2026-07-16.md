# pending 工程4 欠陥: Phase1 ui.spec の状態設定の欠落（2026-07-16）

> `/flywheel` の隔離草案（工程4=翻訳の欠陥 → 上流 PM／AIアーキ）。実装(工程6)では回避せず報告する。
> acceptance は spec/* ブランチでのみ修正可（feature からは read-only）。下記は上流での修正提案。

## 事象

Phase1 の ui.spec を工程4 でスライスごとに独立に書いた結果、**共通の認証・データ状態セットアップ（storageState / 前提データ）を欠き**、スライス間・テスト間で前提が矛盾する。

### 1. slice-06 `auth.ui.spec.ts` test2 と slice-01 が矛盾（緑にできない）
- slice-01: `page.goto('/reports/new')` → 入力画面（textbox 可視）を要求。
- slice-06 test2: `page.goto('/reports/new')` → `/login` へリダイレクトを要求（未認証ガード）。
- どちらも認証状態を設定しない fresh コンテキスト。同じ URL に相反する挙動は両立不能。
- **実装判断（工程6）**: 認証ガードを付けない（slice-01 を壊さない）。slice-06 test2 は**赤のまま**（他は緑）。

### 2. slice-05 `previous.ui.spec.ts` test2 はテスト自体が弱い（偶然緑）
- test2 は「前回の報告はありません」を要求するが、fixture user(staff01) は確定済み報告を持つため前回は必ず存在する。
- 実装は honest（前回があれば表示）。ただし**初期レンダリング（fetch 解決前）の null 状態で一瞬「ありません」が可視**になり、`toBeVisible` がそれを捕まえて偶然緑になる。
- 「前回なし」を検証できていない（ローディング状態を拾っているだけ）。

## 提案（上流 工程4 の修正）

- 全 ui.spec に**認証・データ状態の共通セットアップ**を入れる（Playwright `storageState` かログインステップ、per-test の前提データ）。
  - 認証済み系（01,02,03,04,05,07）は fixture セッションを設定。
  - slice-06 は「未認証コンテキスト」を明示的に用意し、test2 のリダイレクトを正しく検証できるようにする。
  - slice-05 test2 は「前回を持たない別ユーザー」で検証する（staff01 とは別 fixture）。
- 併せて overview/仕様表の「認証セットアップ」を明文化（レジストリの『認証は slice-06 で被せる』を、テスト状態設計まで具体化）。
