# slice-11 project-linking — 確定時の案件キー紐づけと INCIDENTS ステータス

> 振る舞いの正本: `reference-mock/spec.md §3.3`・データモデルは `spec.md §4`（PROJECTS／REPORT_PROJECTS／INCIDENTS）。
> 依存: slice-03（報告確定）。フェーズ: 3。
> ねらい: 報告確定時に対応案件を**案件キー**で既存案件へ紐づけ、後段の突合（slice-12）を決定的にする土台を作る。

## AC-1 報告確定時に対応案件を案件キーで既存案件へ紐づける

- source: reference-mock        # spec.md §3.3「対応案件は確定時に既存案件へ紐づけ（案件キー）でき、これにより後段の突合を決定的に行える」
- **Given** 対応案件（案件キー）を含む下書き報告
- **When** 報告を確定する
- **Then** その案件キーに一致する PROJECTS へ REPORT_PROJECTS の紐づけが作られ、確定報告と案件が対応づく

## AC-2 INCIDENTS は「発生／対応中／解決」のステータスを持つ

- source: reference-mock        # spec.md §3.3「インシデントは『発生／対応中／解決』のステータスを持ち、突合時の状態更新（上書き）に用いる」・§4 INCIDENTS
- **Given** 対応案件にインシデントを含む報告
- **When** 報告を確定する
- **Then** INCIDENTS に `status`（`発生` / `対応中` / `解決` のいずれか）が案件（project_id）に紐づいて保存される

## AC-3 未知の案件キーは新規案件として作成する（突合を決定的にする）

- source: PM        # ★案件キーの解決規則（既存一致／新規作成）は参照モックが明記しない。overview §4 の新規決定
- 理由: 参照モックは「既存案件へ紐づけ」とだけ述べ、未知キー時の挙動を規定しない。PROJECTS は `(user_id, project_key)` を持つ（§4）ため、**同一ユーザー内で案件キーが既存なら既存 PROJECT に紐づけ、未知なら新規 PROJECT を作成**する。これにより後段の突合が案件キー単位で決定的になる。
- **Given** そのユーザーに存在しない案件キーを含む報告
- **When** 報告を確定する
- **Then** 当該ユーザー配下に PROJECTS が1件新規作成され、その案件へ紐づく（既存キーなら重複作成せず既存へ紐づく）

## AC-4 不正なインシデントステータスは 422 で拒否する

- source: PM        # ★バリデーション失敗のコードは参照モックが規定しない。CLAUDE.md §6・overview §3
- 理由: `発生` / `対応中` / `解決` 以外の値は受理しない。バリデーション失敗は **422** を返す（CLAUDE.md §6・500/400 では受け入れテストが赤）。
- **Given** インシデントステータスが列挙値（発生/対応中/解決）以外の報告
- **When** 報告を確定する
- **Then** `422` が返り、紐づけ・案件作成・インシデント保存のいずれも行われない
