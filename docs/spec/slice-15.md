# slice-15 report-cycle-status — 報告サイクル・締切・報告ステータス（5 種）

> 振る舞いの正本: `reference-mock/phase2-design.md §6`（報告サイクルと未報告・報告漏れ管理）。依存: slice-14。
> **重要**: 本スライスは 5 ステータスの**判定セマンティクス**（何が提出済み／遅延提出／未報告／欠勤／報告漏れか）を固定する。
> **REST パス・エンティティ構造・スケジュール（機会）生成の詳細は phase2-design.md §7 に従い実装時の詳細設計とする**（設計凍結ゲート §4・§8 の項目9〜15）。これは凍結ゲートの穴ではなく、answer key（phase2-design.md §7）自身が「実装時に詳細設計」と明記した**明示的な後回し**であり、本スライスは判定セマンティクスと本人閲覧の可否までを固定する。ここでは URL を発明せず、状態の定義と遷移だけを AC 化する。
> **TZ 方針**: 保存 UTC・締切判定はユーザーのローカル時刻（phase2-design.md §6 冒頭・CLAUDE.md 原則9）。
> **決定（source: PM・2026-07-15）**:
> - **本人の履行状況閲覧（項目12・§6.9 未決④）**: スタッフ本人は自分の履行状況を **read-only で閲覧できる**に確定（AC-6・UI-AC）。異議申立ての手段は後続 **slice-25 report-history-appeal で扱う**（PM 決定 2026-07-15）。理由: 本人が自分の記録を確認できることは最小限の透明性であり、閲覧のみなら人事評価の判定ロジックに影響しない。
> - **締切ずらし・遅延累積換算の閾値・報告漏れ計上のばらつき抑止（項目13・14・15）**: 権限3軸・運用ルールに依存するため後続 **slice-24 permission-model で扱う**（PM 決定 2026-07-15）。本スライスは「遅延が累積で報告漏れに換算される」方針の存在のみ確認し、閾値は置かない。
> - **退勤連動型の締切（§6.6・Phase 2 最後段〜Phase 3 級）**: 後続 slice-24／勤怠連携で扱う。本スライスは固定時刻型の締切のみを対象にする。

## AC-1 管理者がスタッフごとに報告サイクルを設定できる

- source: reference-mock        # phase2-design.md §6.1（日報/週報/隔週/月報・スタッフごとに異なる）／エンティティ・REST は ★PM で凍結
- **Given** 管理者と担当スタッフ
- **When** そのスタッフの報告サイクルを 日報／週報／隔週報告／月報 のいずれかに設定する
- **Then** 設定が保存され、スタッフごとに異なるサイクルを持てる（締切時刻＝固定時刻型はスタッフ自身が設定・§6.2）

## AC-2 締切内の確定は「提出済み」、締切後の確定は「遅延提出」になる

- source: reference-mock        # phase2-design.md §6.4（提出済み＝締切内に確定／遅延提出＝締切後に確定）
- **Given** 報告義務が発生している機会（稼働日・承認済み稼働中アカウント・長期休暇でない＝§6.3）と、その締切時刻（ローカル判定）
- **When** スタッフが締切時刻の前／後に報告を確定する
- **Then** 締切前の確定は `提出済み`、締切後の確定は `遅延提出` として記録される

## AC-3 締切後に報告が無ければ「未報告（自動検知の中立状態）」で、評価には効かない

- source: reference-mock        # phase2-design.md §6.4・§6.5（未報告は自動検知の中立状態／システムは自動で評価を下げない）
- **Given** 報告義務が発生している機会で、締切を過ぎても確定報告が存在しない（下書きのままの締切超過も含む・§6.4）
- **When** システムが締切超過を自動検知する
- **Then** その機会は `未報告` になる。これは中立状態であり、この時点では評価に効かない（`未報告` が自動で `報告漏れ` にはならない）

## AC-4 「報告漏れ」は管理者が実態確認の上で計上して初めて確定する

- source: reference-mock        # phase2-design.md §6.5（管理者が実態確認しスタッフ起因と判断して計上・人間の確認を一枚かませる）
- **Given** `未報告` の機会
- **When** 管理者が実態を確認し、スタッフ起因と判断して「報告漏れ」を計上する
- **Then** その機会が `報告漏れ` として記録され、ここで初めて評価に効く（システムが自動で計上することはない）

## AC-5 欠勤はスタッフ申告＋管理者承認で付与され、記録として残る

- source: reference-mock        # phase2-design.md §6.4（欠勤＝申告＋管理者承認による正当な不在／消去せずステータスとして残す）
- **Given** 稼働予定だが不在のスタッフ
- **When** スタッフが欠勤を申告し、管理者が承認する
- **Then** その機会は `欠勤` になり評価対象外となる。欠勤は消去ではなくステータスとして残り、後から経緯を検証できる

## AC-6 本人は自分の履行状況を read-only で閲覧できる

- source: PM        # ★項目12・§6.9 未決④ を PM 決定（2026-07-15）で解消。閲覧のみ・異議申立ては slice-25 へ
- 理由: 本人が自分の記録（提出済み/遅延提出/未報告/欠勤/報告漏れ）を確認できることは最小限の透明性であり、閲覧のみなら人事評価の判定ロジックに影響しない。異議申立ての手段は後続 slice-25 report-history-appeal で扱う。
- **Given** 自分の機会に履行状況（提出済み/遅延提出/未報告/欠勤/報告漏れ）が記録されているスタッフ本人
- **When** 自分の履行状況を参照する
- **Then** 各機会の状態を **read-only** で閲覧できる。閲覧のみで、状態の変更・報告漏れの計上・欠勤の承認などの操作はできない

## 画面要件

- 対象画面: 本人の履行状況閲覧（read-only）。専用の重い画面は新設せず、本人の報告履歴に履行状況を read-only で表示する面として置く。管理者向けのステータス表示は S8（slice-14）のスタッフ一覧の列に載る（本スライスで管理者向け新規画面は足さない）。
- 理由: 本スライスの中心は報告サイクル・締切・5 ステータスの**判定ロジック**。本人閲覧の可否（read-only）はここで確定する（項目12・PM 決定 2026-07-15）が、本人向け履歴画面の DOM 詳細は phase2-design.md §7 に従い実装時の詳細設計とする。
- golden 撮影: 不可（判定セマンティクスが中心・画面詳細は実装時詳細設計）。検証は状態遷移のアサーションと本人閲覧の DOM アサーションで行う。
- **UI-AC（source: PM）**
  - 理由: 参照モックは本人向け履歴画面の DOM・文言を規定しない。本人閲覧の可否をここで確定する。
  - スタッフ本人は自分の履行状況（提出済み/遅延提出/未報告/欠勤/報告漏れ）を read-only で一覧でき、各機会の状態がテキストで表示される（色のみに頼らない）。編集・計上・承認などの操作導線は本人には出さない。

## 合成フィクスチャ（PM 所有）

**本番/実データは一切入れない**（憲法 §1-6）。TZ は保存 UTC・締切判定はローカル。**人事評価の根拠になるデータのため、誤記録が起きない合成値に倒す**（§6 冒頭）。

| フィールド | 階層 | 例 | 備考 |
|---|---|---|---|
| staff.id | L0 | `s_0001` | 報告義務の主体 |
| cycle.type | L0 | `daily` / `weekly` / `biweekly` / `monthly` | 管理者が設定（§6.1） |
| deadline.local | L0 | `18:00`（保存 UTC・判定ローカル） | 固定時刻型のみ（退勤連動は slice-24／勤怠連携へ） |
| opportunity.eligible | L0 | `true`（稼働日・承認済み稼働中・長期休暇でない） | 報告義務の前提（§6.3） |
| status.value | L0 | `submitted` / `late` / `missing` / `absent` / `unreported_flagged` | 5 ステータス（提出済み/遅延提出/未報告/欠勤/報告漏れ） |
| absence.approved_by | L0 | `m_0001` | 欠勤承認者（申告＋承認） |
