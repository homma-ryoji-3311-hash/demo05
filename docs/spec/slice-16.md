# slice-16 slack-reminder — Slack リマインド（短間隔ジョブによる抽出送信）

> 振る舞いの正本: `reference-mock/spec.md §3.8`（通知の実装方式）＋ `phase2-design.md §6.8`（通知チャネルの段階分け）。
> 依存: slice-13（通知設定）・slice-15（未報告ステータス）。
> **TZ 方針**: 保存 UTC・判定はユーザーのローカル時刻（spec.md §3.8・CLAUDE.md 原則9）。
> **段階**: まずバッジ通知（アプリ内・軽い）→ その後 Slack／メール（外部連携・後段）。
> **注意**: リマインド送信は `Summarizer` と同様に**提供元非依存の通知抽象化層**の背後で行い、Slack/メールの SDK・Webhook をドメイン層に書かない（overview §5 の思想を通知に適用）。
> **決定（PM・2026-07-15）**:
> - 外部 Slack/メール送信は受け入れテストで**決定的フェイク notifier に差し替える**（slice-06 の OAuth と同方針）。実 Slack Webhook・実メール送信を CI に持ち込まず、フェイク notifier への「誰に・どのチャネルで送ったか」の送信呼び出しをアサートする。source: PM。理由: 参照モックは外部通知のテスト扱いを規定しない＝新規決定（overview §8.5 凍結解除ログ）。テスト方式の決定のため新規 AC は立てず、AC-1〜4 の検証手段に反映する（下記「画面要件」の golden 撮影欄・合成フィクスチャの `notifier.sink`）。
> - 退勤連動を起点にした締切・リマインドのタイミング（phase2-design.md §6.6・Phase 2 最後段〜Phase 3 級）は本スライスのスコープ外。**後続スライス（Phase 2 最後段〜Phase 3 級）で扱う（PM 決定 2026-07-15）。**

## AC-1 短間隔ジョブが「今この時刻に通知すべきユーザー」を DB から抽出して送る

- source: reference-mock        # spec.md §3.8「全員分のスケジュールを個別に立てず、短間隔ジョブが対象ユーザーを DB から抽出して送信」
- **Given** リマインド時刻（ローカル）を設定した複数のスタッフ
- **When** 短間隔ジョブが実行時刻に走る
- **Then** その時刻に通知すべきユーザーだけを DB から抽出して送信する（ユーザーごとに個別スケジュールを立てない）

## AC-2 リマインド時刻の判定はローカル、保存は UTC で一貫する

- source: reference-mock        # spec.md §3.8「保存を UTC、表示・判定をユーザーのローカル時刻」
- **Given** 異なるタイムゾーンのスタッフが同じローカル時刻（例: `18:00`）を設定している
- **When** ジョブがある UTC 時刻に走る
- **Then** 各ユーザーのローカル時刻に換算して判定し、ローカルで `18:00` を迎えたユーザーだけが抽出される

## AC-3 通知はまずバッジ通知（アプリ内）、Slack/メールは通知設定に従う

- source: reference-mock        # phase2-design.md §6.8（まずバッジ通知）＋ spec.md §3.8（Slack メイン・メールは設定で ON/OFF）
- **Given** バッジ通知に加え、`slack_enabled` / `email_enabled` を設定したスタッフ（slice-13 の通知設定）
- **When** リマインドが発火する
- **Then** アプリ内バッジ通知が作られ、Slack/メールは各ユーザーの有効設定に従ってのみ送られる（OFF のチャネルには送らない）

## AC-4 報告済みのユーザーにはリマインドを送らない（未報告ステータスと連動）

- source: reference-mock        # spec.md §3.8「管理コンソールの未報告ステータスと連動し、誰にリマインドが必要かを可視化」
- **Given** 当日の報告義務がある機会について、確定済み（提出済み）のスタッフと未確定のスタッフ
- **When** リマインドジョブが走る
- **Then** 未提出（未報告になりうる）スタッフのみが対象になり、提出済みのスタッフには送らない

## 画面要件

- 対象画面: なし（バックグラウンドジョブ＋通知チャネル送信であり専用画面を持たない）
- 理由: リマインドはジョブ起点の送信処理で、UI は持たない。通知の ON/OFF・時刻を編集する画面は slice-13（S9 通知設定）が担う。撮影対象の DOM が無いため golden 撮影は不可。
- golden 撮影: 不可。検証は通知抽象化層のフェイクに対する「誰に・どのチャネルで送ったか」のアサーションで行う（実 Slack/メールへは送らない）。

## 合成フィクスチャ（PM 所有）

**本番/実データは一切入れない**（憲法 §1-6）。Slack Webhook URL・Bot トークン等の秘密は `.env` のみで扱い、フィクスチャ・差分に出さない（憲法 §1-7）。TZ は保存 UTC・判定ローカル。

| フィールド | 階層 | 例 | 備考 |
|---|---|---|---|
| userA.timezone | L0 | `Asia/Tokyo` | ローカル判定 |
| userB.timezone | L0 | `Asia/Singapore` | 別 TZ・同一ローカル時刻の弁別 |
| user.remind_time | L0 | `18:00`（ローカル） | 抽出条件 |
| user.slack_enabled | L0 | `true` / `false` | チャネル分岐（slice-13 連動） |
| job.run_at | L0 | `2026-07-15T09:00:00Z` | ジョブ実行時刻（UTC） |
| opportunity.submitted | L0 | `false` | 提出済みは対象外（slice-15 連動） |
| notifier.sink | L0 | `fake-notifier`（送信先を捕捉） | 実送信せず内容をアサート |
