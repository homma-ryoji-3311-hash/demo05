# slice-18 voice-input-stt — 音声入力・STT（テキスト化 → 本文集約・確定前に修正可能）

> 振る舞いの正本: `reference-mock/spec.md §3.2（音声は本文へ追記）・§3.3（STT 2段構成・確定前に修正）・§6.3（degrade）`。
> 依存: slice-01（`POST`/`PATCH /reports`・S3 業務報告入力）。Phase 5。**S3 業務報告入力の拡張。**
>
> **決定（PM・2026-07-15）: STT 方式は Web Speech API（クライアント側）から始める。**
> source: PM。理由: MVP はブラウザ内蔵の Web Speech API でクライアント側完結とし、**差し替え可能な抽象層の裏**に置く（サーバ側 STT は後から足せる形にする。overview §8.5 凍結項目5 の凍結解除）。
> クライアント側完結が既定のため、STT 用の新規サーバ REST エンドポイントは本スライスでは立てない（既存 `POST`/`PATCH /reports` で本文を保存する）。
> AC-1〜3 の観測点は方式非依存のまま有効で、権限拒否・失敗時の degrade（AC-3）は Web Speech でも同様に成立する。

## AC-1 音声入力の結果テキストは本文へ追記され、キーボード入力と併用できる

- source: reference-mock        # spec.md §3.2「自由文モードではキーボード入力と音声入力を同一画面で併用でき、音声は本文へ追記される」
- **Given** 自由文モード（`input_mode` = 自由文）の報告入力で、既にキーボードで入力した本文（`raw_text`）がある
- **When** 音声入力を行い、STT でテキスト化された結果が得られる
- **Then** STT 結果は既存の本文を消さずに**末尾へ追記**され、その後もキーボードで継続して編集できる（本文は編集可能なテキストへ集約される）

## AC-2 STT 結果は確定前に必ずユーザーへ提示され、誤認識を修正できる

- source: reference-mock        # spec.md §3.3「STTの結果は確定前に必ずユーザーへ提示し、誤認識を修正できるようにする」
- **Given** 音声入力による STT 結果テキスト
- **When** その結果を本文へ取り込む（確定する）前
- **Then** STT 結果がユーザーに提示され、本文へ集約する前に**修正できる**（誤認識を直せる）。無条件に自動確定しない

## AC-3 録音権限拒否・STT 失敗時も入力内容を失わない（degrade）

- source: reference-mock        # spec.md §6.3「録音権限拒否・通信断・STT失敗時も入力内容を失わないエラーハンドリング」／状態は色だけでなくテキストでも提示
- **Given** 既に入力済みの本文がある報告入力
- **When** 録音権限が拒否される、または STT がテキスト化に失敗する
- **Then** 既存の本文は**保持**され、キーボード入力を継続できる（degrade で報告は必ず出せる）。失敗は**色だけでなくテキストでも**提示される

## 画面要件

- 対象画面: **S3 業務報告入力の拡張**（`docs/design/overview.md` §2 S3・spec.md §3.2）。音声入力の起動・STT 結果提示・本文追記の UI を S3 に足す。
- golden 撮影: **静的表示（音声入力ボタン・STT 結果の確認/修正欄・失敗時テキスト表示）は撮影可**。音声認識中の動的状態（マイク稼働中など）は golden ではなく **DOM アサーションで検証**（ADR-0008・0018）。
- **UI-AC（source: PM）** — 参照モックに S3 の DOM 詳細が無いため本当の設計。理由: 具体的なセレクタ・文言・操作フローは overview に未記載で、ここで決め切る。
  - 自由文モードの入力画面に**音声入力の起動操作**があり、起動しても既存の本文入力欄は消えず併用できる（AC-1 の UI 表現）。
  - STT 結果は**本文へ取り込む前に確認・修正できる領域**に表示され、ユーザーが取り込みを操作して初めて本文へ追記される（AC-2 の UI 表現。自動確定しない）。
  - 録音権限拒否・STT 失敗時は、入力済み本文を保持したまま**エラーをテキストで**（色のみに依存せず）提示し、キーボード入力を継続できる（AC-3 の UI 表現・アクセシビリティ §6.3）。
  - **STT 方式は Web Speech API（クライアント側）に決定（PM）。** Web Speech の権限拒否・認識失敗時も、入力済み本文を保持したまま degrade で報告は必ず出せる（AC-3 の UI 表現）。認識精度表示・再試行ボタン等の方式依存 UI は差し替え可能な抽象層の裏に置き、本スライスでは上記の方式非依存な観測点を確定する。

## 合成フィクスチャ（PM 所有）

**本番/実データ・音声実データは一切入れない**（憲法 §1-6）。音声バイナリ（実録音）は dev に持ち込まない。STT 結果は合成テキストで代替する。

| フィールド | 階層 | 例 | 備考 |
|---|---|---|---|
| report.input_mode | L0 | `自由文` | 音声はこのモード内でキーボードと併用（spec.md §3.2） |
| report.raw_text（既存本文） | L0 | `"午前は環境構築。"` | 音声追記の前提となるキーボード入力済みテキスト |
| stt.result_text（合成 STT 結果） | L0 | `"午後はレビュー指摘の修正を実施。"` | 実音声を使わず合成テキストで代替。本文末尾へ追記される候補 |
| stt.corrected_text | L0 | `"午後はレビュー指摘の修正を実施した。"` | 確定前にユーザーが誤認識を直した後のテキスト（AC-2） |
| stt.failure_reason | L0 | `permission_denied` / `stt_failed` | degrade 検証用の失敗種別（AC-3）。音声実データは含めない |
