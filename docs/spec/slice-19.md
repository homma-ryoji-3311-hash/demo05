# slice-19 question-template-editor — 設問テンプレート編集

> 振る舞いの正本: `reference-mock/spec.md §3.6`（設問テンプレート管理）。
> REST 表現（パス/メソッド/コード）は参照モックに無い＝設計（`overview.md §3` `GET/PUT/POST /question-sets`）。
> 依存: slice-17。画面: S10 設問テンプレート編集。

## AC-1 グループ単位で設問セットを作成・編集・並べ替えできる

- source: reference-mock        # spec.md §3.6「グループ単位で設問セットを自由に作成・編集・並べ替え」
- **Given** manager が担当するグループ
- **When** 順序付きの設問配列で設問セットを作成・更新する（`POST`/`PUT /question-sets`）
- **Then** `200/201` が返り、設問が指定した並び順で保存され、取得時に同じ順序で返る

## AC-2 各設問が回答形式・必須/任意・役割タグを保持する

- source: reference-mock        # spec.md §3.6「回答形式（短文/長文/選択）・必須/任意・役割タグ」
- **Given** 設問セットの各設問
- **When** 回答形式（短文／長文／選択）・必須/任意・役割タグ（案件キー紐づけ／ステータス付き／スキル抽出／内部のみ）を指定して保存する
- **Then** それぞれが設問に保存され、取得時に返る。役割タグは CONTEXT.md の定義（設問の答えがデータ上どの役割を担うか）に一致する

## AC-3 公開前ガードレールが必須ロール不足を検出し公開をブロックする

- source: PM        # ★必須ロール不足は公開をブロック（警告のみにしない）は overview §8.5 の PM 決定（2026-07-15）。コードは overview §3・CLAUDE.md §6
- 理由: 必須ロールを欠いた設問セットは後段のスキルシート生成を壊すため、警告に留めず公開を拒否する。参照モックはコードを規定しないため `422`（バリデーション失敗）。
- 必須ロールの最小構成（仮初期値・answer key §3.6 準拠）: **案件キー紐づけ ≥ 1・スキル抽出 ≥ 1**。運用実データで調整可能な形にする。
- **Given** 案件キー紐づけ役割またはスキル抽出役割を1つも含まない設問セット
- **When** そのセットを公開しようとする
- **Then** ガードレールが不足を検出し、不足したロールを明示したうえで**公開を拒否する**（`422`。公開状態に遷移しない）

## AC-4 版管理により過去報告を壊さない

- source: reference-mock        # spec.md §3.6「各報告は作成時のテンプレート版を保持。編集は次回報告から反映。下書き→公開で版を切る。過去報告は壊さない」
- **Given** 公開済み設問セット版 v1 で作成された過去報告
- **When** 設問セットを編集し新しい版 v2 を公開する
- **Then** 過去報告は作成時の v1 を保持したまま変わらず、v2 は次回以降の報告にのみ適用される

## 画面要件

- 対象画面: S10 設問テンプレート編集（`docs/design/overview.md §2`。DOM 詳細は参照モックに無い＝設計）
- golden 撮影: 可
- **UI-AC（source: PM）**
  - 理由: 参照モックは画面の DOM・文言を規定しない。編集 UI の要素はここで確定する。
  - 設問の追加・削除・ドラッグ等での並べ替えができる。
  - 各設問行で回答形式（短文/長文/選択）・必須/任意・役割タグを選択できる。
  - 「公開」操作でガードレールを実行し、不足ロールを**テキストで**明示したうえで公開を拒否する（公開状態にならない。色のみに頼らない・非機能要件）。
  - 公開済み版と編集中（下書き）版が区別して表示される。
