# slice-22 group-settings — グループ別設定（設定駆動）

> 振る舞いの正本: `reference-mock/spec.md §2.3`（マルチグループは一元化＋グループ別設定・設定駆動）・
> `reference-mock/report-quality-design.md §7.4`（設定変更は翌日以降の報告に適用）。CLAUDE.md 原則7（マルチグループは設定駆動）。
> 依存: slice-19（グループ別設問セット）。

## AC-1 グループ固有部分が設定データで切り替わる

- source: reference-mock        # spec.md §2.3「システムは1つに一元化。グループ固有部分（設問セット・様式・担当・タブ）だけを切り替える。全体機能は共通」
- **Given** 設問セット・シート様式・担当・タブ表示が異なる2つのグループ A・B
- **When** 各グループに所属するスタッフがシステムを利用する
- **Then** 適用される設問セット・シート様式・タブ表示は所属グループの**設定データ**から解決され、認証・入力・要約・突合・生成などの全体機能は両グループで共通に動く

## AC-2 コードに分岐を埋めず設定で切り替える

- source: PM        # ★原則7「コード改修を伴わない」を受け入れ観測に落とす（参照モックは検証手段を規定しない）
- 理由: spec.md §2.3 は「設定の追加だけで完結しコード改修を伴わない」と述べるが、検証方法は未規定。設定データのみで新グループを利用可能にする、を受け入れ基準として PM が確定する。
- **Given** 既存コードに手を入れず、設定データにグループ C を追加した状態
- **When** グループ C を所属先とする利用者を扱う
- **Then** グループ C の設問セット・様式・タブが解決され利用できる（グループ固有のコード分岐は追加しない）

## AC-3 設定変更は翌日以降の報告に適用され、確定済み過去報告は不変

- source: reference-mock        # report-quality-design §7.4「翌日から効く」・spec.md §3.6（版管理）・原則6（確定後不変）
- **Given** あるグループの設定（設問セット版・様式等）を本日変更する
- **When** 翌日以降に新しい報告が作成される
- **Then** 新しい設定が新規報告に適用される。既に確定済みの過去報告は作成時点の設定を保持したまま変わらない

## AC-4 グループ設定の編集は担当 manager に限定される

- source: reference-mock        # spec.md §1.3・§6.2「管理者の閲覧/操作範囲はグループ単位で限定・バックエンドで強制」
- 補足: 認可失敗の HTTP コードは 🟥 PM（`403`・deny-by-default・overview §3）
- **Given** あるグループの担当でない manager、または staff
- **When** そのグループの設定を編集しようとする
- **Then** `403` が返り、設定は変更されない
- **決定（source: PM・2026-07-15）**:
  - **グループ移管時の過去報告・マスターデータ（項目3・spec.md §9 論点3）**: **履歴として保持**する。移管しても過去の確定報告は**元グループの不変履歴**として残り、移管先へは**新規報告から**引き継がれる（原則6・確定後不変）。理由: 確定報告は不変ログであり、所属変更で過去の記録を書き換えると監査可能性を損なうため。
  - **どの manager がどのグループを編集できるか（担当範囲）の正確な解決（項目9〜14）**: 権限3軸に依存するため後続 **slice-24 permission-model で扱う**（PM 決定 2026-07-15）。本スライスは spec.md §1.3・§6.2 の「グループ単位・バックエンド強制」までを固定する。
  - **設定項目の具体的なスキーマ・既定値・UI 詳細**: 本スライスでは**設定駆動の骨子**（設定データで切替・翌日反映・過去不変）に留め、細部は後続スライスで詰める。

## AC-5 グループ移管後も過去の確定報告は元グループの不変履歴として残る

- source: PM        # ★項目3・spec.md §9 論点3 を PM 決定（2026-07-15）で解消。過去は履歴保持・新規から移管先
- 理由: 確定報告は不変ログ（原則6）であり、所属変更で過去の記録を書き換えると監査可能性を損なう。移管は将来の新規報告にのみ効かせる。
- **Given** スタッフが所属グループ A からグループ B へ移管された
- **When** 移管後に過去（移管前・確定済み）の報告を参照する
- **Then** その過去報告は**元グループ A の不変履歴**として作成時点の設定・様式のまま残る。移管先グループ B の設定が過去報告へ遡って適用されることはなく、B の設定は移管後の新規報告から適用される

## 画面要件

- 対象画面: 管理者コンソール（S8）内のグループ設定編集領域（`docs/design/overview.md §2`。専用画面は overview に無く、管理コンソール内の設定領域として置く）
- 判断の根拠: グループ固有部分の多くは設定データ／API レベルだが、様式・担当・タブ・設問セット版の切替を manager が編集する面が要る。専用画面を新設せず S8 内に置く。
- golden 撮影: 可
- **UI-AC（source: PM）**
  - 理由: 参照モックは画面の DOM・文言を規定しない。設定編集面はここで確定する。
  - グループを選択し、そのグループの設問セット版・シート様式・担当・タブ表示を設定できる。
  - 保存時に「変更は翌日以降の報告に適用される」旨がテキストで示される。
