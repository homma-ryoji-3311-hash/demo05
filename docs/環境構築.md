## 環境構築

要件: Node.js 22+ / pnpm 10+ / PostgreSQL（Docker で動かすか、PCに直接インストールするかは自由）

PostgreSQL の用意の仕方で2通り。どちらか1つを実行すればOK。

### Docker を使う場合

Dockerとは仮想コンテナです。自分のPCの中に、環境に影響されない『独立した小さなパソコン（コンテナ）』を構築できます。 PostgreSQL（データベース）を Docker のコンテナとして動かすことができるため、PostgreSQL 自体を PC にインストールする必要がなくなります。

#### Docker Desktop のインストール（済みの場合はスキップ）

1. [Docker Desktop 公式サイト](https://www.docker.com/products/docker-desktop/) を開き、自分のPCに合ったものをダウンロードする
   - Mac: 「Download for Mac」— Apple チップ（M1〜M4）なら **Apple Silicon**、Intel Mac なら **Intel Chip** を選ぶ（メニュー →「このMacについて」で確認できる）
   - Windows: 「Download for Windows」
2. ダウンロードしたインストーラーを実行する
   - Mac: `Docker.dmg` を開いて Docker をアプリケーションフォルダにドラッグ
   - Windows: `Docker Desktop Installer.exe` を実行（途中で「WSL 2 を使う」の選択肢が出たらそのままチェックを入れてOK。再起動を求められたら再起動する）
3. Docker Desktop を起動する（初回は利用規約への同意が出る。アカウント登録・サインインはスキップしてよい）
4. 画面上部のメニューバー（Mac）/ タスクトレイ（Windows）にクジラのアイコンが出て安定したら準備完了。ターミナルで確認:

   ```sh
   docker --version   # バージョンが表示されればOK
   ```

> 注意: Docker Desktop が起動していない状態で `make setup` を実行するとエラーになる。開発のたびに Docker Desktop を起動しておくこと（自動起動の設定も可能: Settings → General → Start Docker Desktop when you sign in）。

#### セットアップ実行

```sh
make setup   # ツール確認 → pnpm install → .env作成 → DB起動(Docker) → マイグレーション
```

セットアップ後、データベースが動いているかは Docker Desktop の「Containers」画面で `staff-report-db` が Running（緑）になっているかで確認できる。2回目以降の起動・停止は `make db-up` / `make db-down`。

### Docker を使わない場合（PostgreSQLをPCにインストール）

まず PostgreSQL をインストールして起動する（下記「PostgreSQL のインストール」参照）

macOS (Homebrew):

```sh
brew install postgresql@17
brew services start postgresql@17
```

Homebrew 版はパスワードなし・macOS のユーザー名で接続できる状態になるので、そのまま `make setup-local` を実行すればよい（追加の指定は不要）。

Windows: [公式インストーラー](https://www.postgresql.org/download/windows/) をダウンロードして実行（インストール時に設定したパスワードを `make setup-local DB_USER=postgres DB_PASSWORD=<パスワード>` で指定する）。

その後:

```sh
make setup-local   # ツール確認 → pnpm install → .env作成 → DB作成 → マイグレーション
```

接続ユーザー名・パスワードが初期値と違う場合は変数で指定できる:

```sh
make setup-local DB_USER=postgres DB_PASSWORD=postgres
```

## ローカル起動

```sh
pnpm dev          # web (http://localhost:5173) と api (http://localhost:3000) を同時起動
```

web の `/api/*` へのリクエストは Vite の proxy 経由で api に転送される。

## 主なコマンド

| コマンド         | 内容                     |
| ---------------- | ------------------------ |
| `pnpm dev`       | 全アプリの開発サーバ起動 |
| `pnpm build`     | 全アプリのビルド         |
| `pnpm test`      | テスト実行 (Vitest)      |
| `pnpm lint`      | Lint (ESLint)            |
| `pnpm typecheck` | 型チェック               |
| `pnpm format`    | フォーマット (Prettier)  |

## フロントの API クライアント生成

web の API 型・hooks は service の OpenAPI から orval で生成する。生成物（`apps/web/src/common/api/generated/`）はコミット済みなので、clone 直後の `pnpm dev` はそのまま動く。

API の契約（service 側のエンドポイント／スキーマ）を変えたら再生成する:

```sh
make gen-api   # service の openapi.json 出力 → orval で web のクライアント再生成
```

- 契約の単一ソースは各フィーチャーの `contract`（例: `apps/service/src/template/interfaceAdapter/api/contract/greetingContract.ts`）
- `make gen-api` は `apps/service/openapi.json` を出力してから web のクライアントを再生成する
- 生成物は手で編集しない（ESLint でフィーチャー扱いから除外済み）

## データベース

PostgreSQL 17 + Prisma。接続先は `apps/service/.env` の `DATABASE_URL` で切り替える（Docker: `localhost:5433` / ローカルインストール: `localhost:5432`）。

- スキーマ定義: `apps/service/prisma/schema.prisma`
- 生成クライアントは `apps/service/src/generated/`（gitignore済み。clone直後は `db:generate` が必要）
- 本番コード（`main.ts`）は `PrismaReportRepository`、テストは `InMemoryReportRepository` を注入する
- Prismaクライアントのimportは `infrastructure/` とコンポジションルートのみ許可（ESLintで強制）

DB操作コマンド（`pnpm --filter @staff-report/api <cmd>` または apps/service 内で実行）:

| コマンド      | 内容                                       |
| ------------- | ------------------------------------------ |
| `db:migrate`  | マイグレーション作成・適用（開発用）       |
| `db:deploy`   | マイグレーション適用のみ（本番用）         |
| `db:generate` | Prismaクライアント生成（`src/generated/`） |
| `db:studio`   | Prisma Studio（DBのGUI）起動               |
