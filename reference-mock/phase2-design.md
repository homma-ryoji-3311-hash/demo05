# Repo Base — Phase 2 以降 設計方針

## 0. このドキュメントについて

- 本書は Repo Base（業務報告・スキルシート生成システム / フロンティアダイレクト）の **Phase 2 以降** で実装する、権限・承認・報告サイクル管理の設計方針をまとめたもの。
- **Phase 1（現行）では実装しない。** Phase 1 は「自分の報告のみ／deny-by-default」の最小権限で動作する。本書は将来の実装指針であり、Phase 1 のコードを縛るものではない。
- 各項目は「決定事項」と「未決事項（要検討）」を区別して記す。未決事項は実装着手時に詰める。
- Phase 1 で構築した認可の要 `get_owned_report`（全報告アクセスが必ず通る所有者検証）を、本書の権限モデルへ拡張する形で発展させる。Phase 1 の「全アクセスは関所を必ず通る」設計が土台。

## 1. 権限モデル：3つの独立した軸

権限は単一の役割名ではなく、3つの独立した軸の組み合わせで決まる。「役割」と「担当範囲」と「機微情報アクセス」を混同しないことが設計の肝。

### 軸1：操作権限レベル（何ができるか）
- system admin（情シス／社内SE層）：システムの土台を管理。最上位アカウントの管理・付け替え。
- super admin（中央管理者／マスター）：管理者の任命、チャネルの割り当て。
- manager（管理者）：担当スタッフの報告閲覧・承認・担当割り当て 等。
- staff（スタッフ）：自分の報告の入力・閲覧のみ。
- 未決事項①：system admin と super admin を別の段として分けるか、統合して1段にするか。分ける利点＝マスターが事故っても情シスが復旧できる（堅牢）。統合の利点＝単純（最上位アカウント管理は運用に委ねる）。

### 軸2：担当範囲（どのスタッフを見られるか）
- スタッフと管理者は多対多の担当関係で結ぶ。チャネル単位ではなく個人単位で割り当てる。
- 同じチャネルでも、自分の担当でないスタッフの情報は見えない（営業担当＝管理者となるケースを想定し、担当外の機微情報を遮断する）。
- 担当には主担当（メイン）／副担当（サブ）の区別がある（→ 第3章）。

### 軸3：機微情報アクセス（雑感を見られるか）
- 雑感（メンタル面の記録）の閲覧可否は、担当関係とは別レイヤーで制御する。「担当だから雑感も見える」ではない（→ 第4章）。

### メンタルケア担当（独立役割）
- 全スタッフの全情報（雑感を含む）を閲覧できる。
- ただし権限付与・承認・担当割り当て等の操作権限は一切持たない（閲覧特化）。
- super admin（操作権限が最大）とは性質が逆（可視範囲は最大・操作は最小）。明確に別役割として扱い、「全部見えて全部操作できる最強アカウント」を作らない。

## 2. チャネル
- HR / SES / マーケ 等の事業領域区分。
- 管理者は複数チャネルを掛け持ちできる（例：HR+SES、HR+マーケ、全部）。

## 3. 主担当／副担当
スタッフ×管理者の担当関係に「主担当か副担当か」の属性を持たせる。

| 操作 | 主担当（メイン） | 副担当（サブ） |
|---|---|---|
| 業務報告の閲覧 | ○ | ○ |
| 雑感の閲覧 | ○ | × |
| 承認 | ○ | × |
| 担当割り当て | ○ | × |
| スキルシート生成 等の一部操作 | ○ | ○（雑感・承認を除く） |

- 1人のスタッフに、主担当・副担当を含む複数の管理者を紐付けられる。
- 方針：主担当＝フル権限／副担当＝業務報告の閲覧＋一部操作（雑感の閲覧と承認は不可）。
- 未決事項②：副担当に許す操作の正確な線引きは、Phase 2 で各機能を実装する際に画面を見ながら確定する。

## 4. 雑感（メンタル面の記録）の扱い
- 記入：全チャネルの全スタッフが、自分の雑感を記入できる（部門を問わない）。
- 閲覧：本人（自分の雑感）／そのスタッフの主担当管理者／メンタルケア担当（全スタッフ分）のみ。
- 副担当・一般の管理者・super admin は、雑感を閲覧できない。
- 雑感は AI による専門用語化・スキルシート変換の対象から完全に除外する（Phase 1 からの一貫方針）。

## 5. 登録・承認フロー
- 新規スタッフは Google ログイン後、承認されるまで「承認待ち」画面で止まる。報告入力を含め何もできない（deny-by-default）。
- 管理者が承認を行い、その際に担当の紐付け（主担当／副担当）・チャネル・報告サイクルも設定する。
- 管理者画面に「承認待ちスタッフ一覧」を置き、未承認者に気づけるようにする。
- 将来的に、能動的アラート（メール／Slack 等）の追加を検討。
- 未決事項③：承認待ち一覧を誰に見せるか（担当予定の管理者か、super admin か、両方か）。新規スタッフはまだ誰の担当でもないため、「誰が承認すべきか」をどう決めるかも併せて検討。

## 6. 報告サイクルと未報告・報告漏れ管理（評価指標の基盤）
報告の履行状況をスタッフ評価の定量指標に用いることを前提に設計する。最優先事項：このデータは人事評価の根拠になるため、誤記録が不当な評価につながらないよう、データの正確さを最優先する。タイムゾーンは保存 UTC・判定／表示はローカル（CLAUDE.md 原則9）。

### 6.1 報告サイクル設定
- スタッフごとに報告頻度を管理者が設定：日報（毎日）／週報（週1）／隔週報告（2週に1度）／月報（月1）。
- スタッフごとに異なるサイクルを設定できる。

### 6.2 締切時刻（2タイプ）
- 固定時刻型：スタッフ自身が締切時刻を設定（例：毎日 18:00）。
- 退勤連動型：退勤打刻を起点に「退勤から○時間以内」（→ 6.6）。退勤連動型でも、保険の固定締切を必ず併置する（打刻が無くても必ず締切が訪れる）。
- 役割分担：サイクル（日報/週報/…）＝管理者が設定／その日の締切時刻＝スタッフが設定。
- 未決事項⑤：スタッフが締切時刻を後からずらせるか（ずらせると締切逃れに悪用されうる）。

### 6.3 報告義務の前提（未報告判定の土台）
未報告を判定する前に、「そもそも報告義務が発生している日・状態か」を判定する：稼働日か（休日・非稼働日は対象外）／承認済み・稼働中アカウントか（承認待ち・休職中等は対象外）／長期休暇期間でないか（連続有給・産休育休等）。

### 6.4 報告ステータス（5つ）

| ステータス | 意味 | 評価への影響 |
|---|---|---|
| 提出済み | 締切内に確定された | 良好 |
| 遅延提出 | 締切後に確定された（出してはいる） | 累積で報告漏れに換算（→ 6.7） |
| 未報告 | 締切後、報告が存在しない（自動検知する中立状態） | この時点では評価に効かない |
| 欠勤 | スタッフの申告（＋管理者承認）による正当な不在 | 対象外 |
| 報告漏れ | 管理者が実態確認の上スタッフ起因と判断し計上した記録 | 評価に効く |

- 下書きのまま締切超過は「未報告」扱い。ただし締切前に「下書きのままですよ」警告をスタッフに出す（取りこぼし防止）。
- 欠勤は「消去」ではなく「欠勤ステータス」として残す。スタッフ側からの欠勤申告（＋管理者承認）で付与し、後から経緯を検証できるようにする（管理者がログを消去する方式は、検証不能になるため非採用）。

### 6.5 未報告 → 報告漏れ の流れ（二段構え）
締切超過（＝未報告：自動検知）→ アラート1回目（バッジ通知。将来 Slack/メール、スタッフ宛＋管理者宛）→ ○時間後 アラート2回目（同上）→ 管理者がスタッフに連絡し実態を確認 → スタッフ起因と判断 → 管理者が「報告漏れ」を計上（ここで初めて評価に効く）。
- 「未報告（自動検知）」と「報告漏れ（管理者が計上して確定）」は別概念。システムは自動で評価を下げず、必ず人間の確認を一枚かませる。
- 未決事項⑦：報告漏れ計上の管理者間のばらつき・取りこぼしを抑える仕掛け（計上ガイドライン、未報告累積の super admin 可視化 等）。

### 6.6 退勤連動・勤怠サービス連携（Phase 2 最後段〜Phase 3 級）
- 位置づけ：報告サイクル・ステータス管理の土台ができてから乗せる応用機能。
- 退勤打刻を締切の起点に使える（残業時は遅く、早上がり時は早く、締切が自然に動く）。
- 退勤打刻の検知方式（調査結果：2026年6月時点）：ジョブカン勤怠管理は「打刻のリアルタイム Webhook」を標準提供していない。現実的な経路＝(1) 公式 API で定期取得（ポーリング／基本線）、(2) iPaaS（Zapier / Yoom 等）を挟む、(3) ジョブカン標準の Slack 連携打刻を運用で活用。ポーリングは検知に遅延（数分）が出る前提で設計する。実装着手の直前に、各サービスの最新 API/Webhook 仕様を公式ドキュメントで再確認する（仕様は変わりうる）。
- 抽象化：勤怠連携は差し替え可能な部品として抽象化層の裏に隠す（freee / ジョブカン等の特定サービスにロックインしない）。AI 提供元非依存（CLAUDE.md 原則2）と同じ思想。
- 依存リスクの明記：退勤連動は「便利な起点」にとどめ、報告義務の根拠は自システム内で完結させる。外部サービス（勤怠／iPaaS）の障害・仕様変更・乗り換えに評価データが振り回されない設計を保つ。
- 退勤打刻なしの揺らぎ対策：退勤打刻が無い日は、報告義務が宙に浮かないよう保険の固定締切（6.2）が下支えする。「退勤打刻なし」自体は「打刻漏れ」として報告の問題とは切り分ける（勤怠システムの責任範囲）。

### 6.7 遅延提出の累積換算
- 遅延提出が累積すると報告漏れに換算する。換算閾値（例：N 連続、M 回／期間）は設定可能とする。具体値（例：3連続で1、30日以内に5回で1 等）は Phase 2 実装時／運用開始後に調整。
- 未決事項⑥：換算閾値の具体値。

### 6.8 通知チャネルの段階分け
- まずバッジ通知（アプリ内）から実装（軽い）。Slack / メール通知（外部連携・スタッフ宛＋管理者宛）は Phase 2 の後段（重いので後回し）。

### 6.9 評価指標としての利用
- 提出済み／遅延提出／未報告／欠勤／報告漏れ の回数等を、スタッフ評価の定量指標に用いる。
- 未決事項④（透明性）：本人が自分の履行状況（未報告・欠勤の記録）を閲覧できるか、誤記録に異議を申し立てる手段を設けるか。評価の公正さ・納得感の観点で検討。（UI モックでは「報告履歴・履行状況」画面として本人閲覧を想定済み。）

## 7. データ構造に関する設計メモ（Phase 2 実装テーマ）
- 現 reports テーブルの status（draft / confirmed）では、「報告が存在しない状態（未報告／欠勤）」を表現できない。
- 「報告すべき機会（スケジュール）」を管理する構造が必要。サイクル設定から「いつ・誰が報告すべきか」を生成し、各機会に対して 提出済み／遅延提出／未報告／欠勤／報告漏れ を紐付ける。
- 想定される新規エンティティ（実装時に詳細設計）：チャネル／スタッフ×管理者の担当関係（主・副の属性を持つ）／報告スケジュール（機会）とその履行ステータス／欠勤申告／勤怠連携の打刻イベント（抽象化層の裏）。

## 8. 未決事項一覧（実装着手時に詰める）
1. system admin と super admin を分けるか統合するか（第1章）
2. 副担当に許す操作の正確な線引き（第3章）
3. 承認待ち一覧を誰に見せるか／新規スタッフを誰が承認するか（第5章）
4. 本人が自分の履行状況を閲覧できるか・異議申立ての手段（6.9）
5. スタッフが締切時刻を後からずらせるか（6.2）
6. 遅延提出の累積換算の具体的な閾値（6.7）
7. 報告漏れ計上の管理者間ばらつき・取りこぼしを抑える仕掛け（6.5）
