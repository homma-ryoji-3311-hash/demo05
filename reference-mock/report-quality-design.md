# 報告品質向上：AI追加質問機能 設計方針

> このドキュメントは「設計のみ」です。実装はまだ行いません。
> 本文では **【決定】**（このプロジェクトで合意済みの方針）と **【未決→12章】**（まだ決めていない、本文着手時に詰めること）を区別して書きます。
> 「決まったかのように書かない」ことを徹底し、迷う箇所は未決として残します。

---

## 0. このドキュメントについて

- **目的**：粒度（＝報告の細かさ・具体性）が低い項目に対して、AIが一度だけ追加質問を行い、報告の質を底上げする機能の設計方針を定める。
- **スコープ**：設計のみ。コードは書かない。既存コードには触れない。
- **読者**：本プロジェクトの開発者・管理者（運用設計をする人）。初心者にも読めるよう、専門用語には一言の補足を付ける。
- **関連資料**：
  - `CLAUDE.md`（アーキテクチャの重要原則。特に原則2・3・6・7）
  - `docs/spec.md`（要件定義／設計仕様）
  - `docs/phase1-plan.md`（現フェーズの範囲）
  - `docs/phase2-design.md`（権限3軸・グループ設定などの将来方針）

> **用語の最小セット**（先に押さえておくと読みやすい）
> - **報告本文（raw_text）**：スタッフが書いた生のテキスト。後述の原則6で「確定後は変えない」対象。
> - **要約（ai_summary_json）**：AIが報告本文を `incidents / achievements / issues / skills` の4カテゴリの箇条書きに整理したもの。
> - **粒度**：報告の具体性。「対応した」だけは粒度が低い、「○○の手順で△△を設定し××を確認」は粒度が高い、というイメージ。
> - **fallback（フォールバック）**：本来の処理が失敗したときに切り替える「代わりの安全な動き」。

## 1. 機能概要と狙い

### 1.1 解決したい課題（粒度の低い報告）

- 報告が「対応しました」「特になし」のように薄いと、要約・スキルシート・突合の材料にならない。
- 後から本人に聞き直すのは手間がかかり、記憶も薄れる。**書いた直後に、薄い箇所だけ補ってもらう**のが最も低コスト。

### 1.2 ひとことでの動作（要約後に一度だけ追加質問）

- 【決定】**タイミング**：要約を作った**後**に、薄いと判定された項目へ追加質問する。
- 【決定】**対話は一度だけ**：往復を繰り返さない。1回質問し、回答（または無回答）を受けて先へ進む。

### 1.3 やらないこと（非ゴール）

- 【決定】チャットのような多段の往復はしない（尋問・詰問にしない）。
- 【決定】AIに事実や数値を創作させない（原則3）。質問はあくまで**本人に書いてもらう**ためのもの。
- 【未決→12章】音声入力との組み合わせ（STT経由の回答）は本機能の初期スコープに含めるか。

## 2. 全体フロー（最終形）

### 2.1 パイプライン位置

報告が確定するまでの流れに、太字の3ステップを差し込む。

```
報告入力（下書き）
   ↓
要約（Gemini）              ← 既存
   ↓
粒度判定（ルール＋Gemini）   ← ★追加：薄い項目を見つける
   ↓
追加質問（一度だけ）         ← ★追加：薄い項目について本人に質問
   ↓
本文へ追記 → 要約を作り直す   ← ★追加：回答を反映して要約を再生成
   ↓
確認・確定（確定後は不変）     ← 既存
```

> 「パイプライン」＝処理を順に通す流れ作業のこと。

### 2.2 「対話は一度だけ」の境界定義

- 【決定】1つの報告につき、追加質問のラウンドは**1回**。
- 【決定】そのラウンドで複数項目に質問してよい（例：issues と achievements が薄ければ両方まとめて1回で聞く）。
- 【未決→12章】「再生成した要約がまだ薄い」場合に、もう一度だけ聞く例外を将来許すか。初期は**許さない（1回で打ち切り）**を既定とする。

### 2.3 シーケンス図（テキストで）

```
スタッフ        フロント         バックエンド        Summarizer抽象化層
  |  報告を書く     |                |                     |
  |--------------->| 要約して        |                     |
  |                |--------------->| 要約を依頼          |
  |                |                |-------------------->| (Gemini)
  |                |                |<--------------------|
  |                |   粒度判定（ルール→Geminiの2段）       |
  |                |                |-------------------->| (Gemini:吟味)
  |                |                |<--------------------|
  |   薄い項目あり → 追加質問を表示  |                     |
  |<---------------|                |                     |
  |  回答 or スキップ|                |                     |
  |--------------->| 本文へ追記      |                     |
  |                |--------------->| 要約を作り直す      |
  |                |                |-------------------->| (Gemini)
  |                |                |<--------------------|
  |   確認画面 → 確定（不変）        |                     |
```

> 「Summarizer抽象化層」＝AIプロバイダ（Gemini等）を直接呼ばず、差し替え可能な共通の窓口にまとめた層（原則2）。図のAI呼び出しはすべてこの層を通る。

## 3. 粒度判定（対象カテゴリ＋ルール検出）

- 【決定】判定の**引き金（主役）は「管理者が選んだ対象カテゴリ」×「ルールでの検出（空・極端に短い）」**とする。対象カテゴリに該当し、かつルールで薄いと検出された項目を「薄い項目」とする。
- 【決定】**Geminiによる報告ごとの薄さ自動判定は主役にしない**。理由：報告ごとに「薄い／薄くない」をその場で正しく自動判定するのは難しく（妥当な線引きは数日分を見ないと分からない）、誤判定がスタッフ体験を損なうため。Gemini吟味は任意・後続フェーズの補助に位置づける（3.2）。
- 補足：従来「二段方式」と表現していたが、主役はルール検出であり、Gemini吟味は任意の補助段に位置づけ直した（下記）。

### 3.1 主役：対象カテゴリ × ルール検出（決定的）

- 【決定】**管理者が選んだ対象カテゴリだけ**を見る。対象外のカテゴリはそもそも判定・質問の対象にしない。
- 【決定】対象カテゴリの項目について、コードだけで判定できる薄さ（空、空白のみ、極端に短い、定型の無内容＝「特になし」等）をルールで検出する。
- 【決定】**決定的**（同じ入力なら必ず同じ結果。AIのような揺らぎがない）であること。テストしやすく、AI費用もかからない。対象カテゴリは**設定値として読む**（コードに分岐を埋めない＝原則7）。
- 【未決→12章】具体的なしきい値（最小文字数、無内容とみなす語の一覧）と、対象カテゴリの仮初期値（どのカテゴリを対象にするか）。初期値は本文着手時に決める。

### 3.2 補助段：Gemini による中身の吟味（任意・後続フェーズ）

- 【決定】Geminiによる薄さの吟味は**主役ではなく、後続フェーズの任意の補助**に位置づける。最初の一歩（スライス1〜4）には含めない。
- 【決定】将来導入する場合も、**対象カテゴリに限って**ルール検出を補う用途に限る（全カテゴリの自動判定はしない）。
- 【決定】導入時もこの呼び出しは**抽象化層の背後**で行い（原則2）、出力は**固定スキーマのJSON**にする（原則3）。プロバイダ名はドメイン層に出さない。
- 【未決→後続】吟味の判定基準（どの程度を「薄い」とするか）のプロンプト表現。導入可否そのものも後続で判断する。

### 3.3 各段の責務分担と「薄い項目」の定義

| 段 | 役割 | 何を見るか | 位置づけ |
|---|---|---|---|
| 主役（対象カテゴリ×ルール） | 引き金 | 対象カテゴリの形（空・短さ等） | 最初の一歩で実装（決定的・無料） |
| 補助（Gemini吟味） | 任意の上乗せ | 中身の具体性 | 後続フェーズ（任意） |

- 【決定】最初の一歩での「薄い項目」＝**管理者が選んだ対象カテゴリ**に属し、かつ**ルール検出で薄い（空・極端に短い等）**と判定された項目。
- 【決定】対象指定の**単位はカテゴリ**（incidents / achievements / issues / skills のいずれか）とする。理由：要約の各項目はAIが報告ごとに作り直すため固定の目印が無く翌日以降に引き継げないが、カテゴリは毎日同じ枠なので「持続するルール」にできる。**項目単位の指定は後続フェーズの別機能**とする。

### 3.4 しきい値の厳しさをどちらに倒すか

> しきい値＝「これより短い／薄い」と判定する基準。引き金はルール検出（決定的・揺らがない）なので、論点は「誤判定をどう減らすか」ではなく「**基準をきつめ／ゆるめのどちらに倒すか**」になる。
> - **きつくする**（例：必要な文字数を増やす）と、薄い判定が増え → 質問が増える。
> - **ゆるくする**と、薄い判定が減り → 質問が減る。

- 【決定】初期は**ゆるめ（迷ったら質問しない寄り）から始め、実データを見て調整**する。理由：過剰な質問はスタッフの負担・離脱を招くため。追加質問は「付加価値」であって必須経路ではない（10章に接続）。
- 【決定】調整は実環境での運用後に行う（12.5 のスライス4で本物のGeminiに報告を流し、使い心地としきい値を合わせる）。
- 【未決→12章】具体的なしきい値（最小文字数など）の数値。
- （参考）従来は「偽陽性／偽陰性」（本当は十分なのに薄いと誤判定／本当は薄いのに見逃し）という、AIの自動判定を前提にした枠組みで書いていた。引き金をルール検出に変えたため、この構えに置き換えた。なお Gemini 吟味を後続で補助段（3.2）として導入する場合は、中身の自動判定が入るため誤判定の論点が再び関わってくる。

## 4. 追加質問の生成

### 4.1 基本：Gemini が薄い項目を見て生成

- 【決定】基本は Gemini が、薄い項目の内容を見て質問文を生成する（抽象化層経由・固定スキーマ）。
- 【決定】質問は**本人に書いてもらう誘導**であり、答えを示唆・捏造しない（原則3）。

### 4.2 管理者の定型質問の差し込み

- 【決定】管理者が用意した**定型質問**（固定文）を差し込めるようにする。
- 【未決→12章】定型質問を「どのカテゴリに・どの条件で」紐づけるかの設定形。

### 4.3 生成質問と定型質問の優先順位・併用ルール

- 【未決→12章】両方が該当したときの優先順位・併用可否・重複排除の規則。初期は単純な規則（例：定型優先、または片方のみ）から始める想定だが、確定は本文着手時。

### 4.4 質問数の上限・対象項目の選び方

- 【決定】「一度だけ」の趣旨に沿い、**質問総数に上限**を設ける（負担を抑える）。
- 【未決→12章】上限の具体数、複数項目が薄いときの優先順位。

## 5. 回答の扱い（原則3・6との整合の核）

この章は CLAUDE.md 原則3（創作させない）・原則6（確定後不変・増分）と最も密接に関わる。

### 5.1 回答は報告本文へ追記する

- 【決定】追加質問への回答は、要約にではなく**報告本文（raw_text）に追記**する。
  - 理由：要約は「本文から導かれるもの」。本文を真実の源（出どころ）に保ち、要約は常に本文から作り直せる状態にするため。
- 【決定】追記は**確定前の下書き段階**で行う。確定済み報告の本文は変えない（5.3）。
- 【未決→12章】追記の体裁（元の本文との区切り方、質問文も一緒に残すか、追記であることが分かる印を付けるか）。

### 5.2 要約の作り直し（再生成）の手順

- 【決定】本文へ追記した後、**要約を作り直す**（古い要約を更新）。回答が要約に反映された状態で確認・確定に進む。
- 【決定】再生成も抽象化層経由・固定スキーマ（原則2・3）。
- 【決定】これは「全履歴を毎回再処理」ではなく**その報告1件の中での作り直し**であり、原則6の増分マージ（案件キー単位の突合）とは別レイヤーの話。突合は後続フェーズ。

### 5.3 確定後不変（原則6）との関係

- 【決定】**確定（status=confirmed）した報告の本文・要約は変えない**。追加質問・追記・再生成はすべて**確定前**にのみ起きる。
- 【決定】したがって本機能は原則6を破らない。「一度確定したら凍結」という不変条件はそのまま維持される。
- 【未決→12章】確定後に「やはり追記したい」需要が出た場合の扱い（新しい報告として足すのか、本機能の対象外とするのか）。初期は**対象外**を既定とする。

### 5.4 AIに事実を創作させない（原則3）

- 【決定】回答が無い項目は**空欄のまま**にする。AIに埋めさせない。
- 【決定】要約・追加質問のAI出力はすべて**アンカー/フィールド対応の固定JSONスキーマ**に縛る。マスターに無い数値・事実を出させない。

## 6. 必須／任意の制御

### 6.1 回答必須・任意の設定（管理者）

- 【決定】各追加質問（または項目）について、回答が**必須か任意か**を管理者が設定できる。

### 6.2 必須未回答時の挙動

- 【決定】**必須**の追加質問が未回答のときは、**確定をブロックする**（答えるまで確定できない）。
- 【決定】**任意**の質問は**スキップして確定できる**。
- 【決定】**degrade との切り分け（10章と矛盾させない）**：必須ブロックが発動するのは「**質問が提示されたのに、その必須を未回答**」のときに**限る**。
  - AIが失敗して**追加質問自体が出せなかった**場合（10.2・10.3 の fallback）は、**必須ブロックは発動せず**、通常どおり確定できる。
  - 理由：必須は「問われたのに答えていない」状態に課す制約であって、「そもそも問われていない」状態を確定不能にしてはならない（原則：機能が落ちても報告は必ず出せる＝10.1）。

### 6.3 スキップ・無回答の記録

- 【決定】スキップ／無回答でも報告は出せる（10章の degrade 思想と一致）。
- 【未決→12章】無回答だった事実を記録・可視化するか（誰がいつスキップしたか等）。プライバシー・評価利用との兼ね合いは phase2-design.md の評価指標方針と整合を取る。

## 7. 基準の設定モデル（最終形 ↔ 最初の一歩）

「粒度の基準」をどの単位で設定できるか、を段階的に広げる。

### 7.1 最終形：グループ/チームごとの基準（設定駆動・原則7）

- 【決定（ゴール）】最終的には**グループ/チームごと**に基準を設定できる。
- 【決定】コード分岐を増やすのではなく、**設定データで切り替える**（原則7：マルチグループは設定駆動）。

### 7.2 最初の一歩：クライアント全体で単一基準

- 【決定】最初の実装は**クライアント全体で1つの基準**から始める（段階的導入）。
- 【決定】最初からグループ別を作り込まない。まず単一基準で機能全体を通す。

### 7.3 単一 → グループ別へ広げる移行設計（後方互換）

- 【決定（方針）】単一基準を「全体に効くデフォルト1件」として持ち、後でグループ別を**上書き層**として足せる構造を初期から想定する（作るのは後でも、データの形が破綻しないようにする）。
- 【未決→12章】基準の解決順（グループ設定があればそれ、無ければ全体デフォルト、の優先順位の正式定義）。

### 7.4 設定項目の一覧

- 【決定】管理者が設定する対象は最低限：①**対象カテゴリ（どのカテゴリを判定・質問の対象にするか）**、②粒度のしきい値、③各質問の必須/任意、④定型質問、⑤機能の有効/無効。
- 【決定】**対象カテゴリの単位はカテゴリ**（項目単位は後続フェーズ＝3.3）。最初の一歩（スライス1〜4）は**仮の初期値（単一の設定値）**として持ち、管理者が画面で付け替えられるようにするのは土台スライス（12.5 の5番目）で足す。
- 【決定】**「翌日から効く」**：管理者が対象カテゴリ等を変更すると、**翌日以降の報告に自動で適用**される。確定済みの過去報告は不変（原則6）なので遡って変えない。
- 【未決→12章】各項目の具体的なデータ型・既定値・UI。

## 8. AI抽象化層との整合（原則2）

### 8.1 粒度吟味・質問生成も Summarizer 抽象化層の背後で呼ぶ

- 【決定】本機能で増えるAI呼び出し（②粒度の吟味、③質問生成、④要約の再生成）は、**すべて既存の抽象化層の背後**で行う。
- 【決定】ドメイン層（services）に Gemini / Vertex などの具体プロバイダ名やSDKを書かない。

### 8.2 プロバイダ非依存にするためのインターフェース方針

- 【決定】**既存の Summarizer 抽象化層を拡張する**（新しい抽象点は新設しない）。粒度の吟味・質問文の生成を、その抽象化層に**新しい能力として足す**。
- 【決定】Gemini 固有の事情（モデル名・`response_schema` の制約など）は **`gemini.py` に隔離したまま**にする（原則2）。新能力でも具体プロバイダ依存をドメイン層へ漏らさない。
- 理由：すでに動いている仕組みで原則2（提供元非依存）を満たせ、配線（DI・設定解決・例外変換）の二重化を避けられる。

### 8.3 出力スキーマの固定

- 【決定】粒度判定の結果、生成質問は**固定JSONスキーマ**で受け取る（自由文を解釈しない）。
- 【決定】プロバイダ固有のスキーマ制約（例：Gemini の `response_schema` が一部キーを受け付けない等）は**プロバイダ実装側で吸収**し、共通スキーマ定義は汚さない（原則2。昨日の知見＝10.8に接続）。

## 9. データモデルへの影響（設計メモ）

> ここは「設計メモ」。具体的なテーブル列名・型は **【未決→12章】** とし、決まったかのように書かない。

### 9.1 報告本文・要約・追加質問/回答の保持方法

- 【決定（方針）】回答は本文（raw_text）へ追記し、要約（ai_summary_json）は作り直す（5章）。本文を出どころとして一貫させる。
- 【未決→12章】追加質問と回答そのものを別途保存するか（監査・再現のため）。保存するなら格納先。

### 9.2 基準設定・定型質問の格納先

- 【未決→12章】単一基準（初期）とグループ別基準（最終形）をどのテーブル/設定で持つか。7.3 の上書き層と整合させる。

### 9.3 一度きり対話の状態管理

- 【決定】「この報告は追加質問を済ませたか」を表す**状態**が必要（二重質問を防ぐため）。
- 【未決→12章】状態の持ち方（報告に質問済みフラグを足すか、別の状態を持つか）。10.5・10.6の復帰/再試行と整合させる。

## 10. 失敗・例外時の挙動（degrade 設計）

> 「degrade（デグレード）」＝機能の一部が落ちても、致命的にせず**段階的に質を落として動かし続ける**こと。
> この章は、昨日の実環境テストで要約呼び出しが失敗した経験を踏まえて足した。

### 10.1 設計原則：機能が落ちても報告は必ず出せる

- 【決定】追加質問は**付加価値であって必須経路ではない**。AI側が失敗しても、スタッフが報告を確定できる道を必ず残す。
- 【決定】「AIが落ちる＝報告が出せない」は**禁止**。失敗時は追加質問を**省略**して通常フローへ戻す。

### 10.2 Gemini 失敗時の fallback（粒度の吟味が失敗：後続フェーズ）

- 【決定】後続フェーズで補助段（Gemini吟味＝3.2）を導入した場合、その吟味が失敗したら**吟味をスキップ**し、追加質問なしで通常の確認・確定フローへ戻す。
- 【決定】ルール検出（主役・決定的）の足切り結果は決定的なので、吟味が落ちてもそこまでの判定は無駄にしない（必要ならルール検出だけで軽い質問に留める案は 10.4）。

### 10.3 Gemini 失敗時の fallback（質問生成が失敗：最初の一歩）

- 【決定】薄い項目は見つかったが質問文の生成が失敗した場合、**追加質問なし**で確定フローへ戻す。
- 【未決→12章】このとき管理者の**定型質問だけは出す**という縮退案を採るか（生成に依存しない分、出せる可能性がある）。初期は「定型があれば定型のみ、無ければ質問なし」をたたき台とする。

### 10.4 どの段が落ちたかによる切り分け

- 【決定】ルール検出（主役・決定的・無料）は基本的に落ちない前提。落ちうるのは Gemini を使う段（質問生成、および後続で導入する場合の吟味）。
- 【決定】**落ちた段だけを切り離す**。吟味が落ちても生成が要らない場面（定型質問のみ）は動かせる、という形で「全部かゼロか」にしない。

### 10.5 スタッフの途中離脱・未回答のまま再来訪したときの状態

- 【決定】スタッフが追加質問の途中で離脱しても、**報告は下書きとして保持**され、失われない。
- 【決定】再来訪時の既定：**もう一度同じ質問で詰まらせない**。すでに一度提示した質問は「未回答のまま先へ進める（スキップ可能）」状態で見せる。「一度だけ」の約束を、離脱をまたいでも守る。
- 【未決→12章】再来訪時に「未回答の質問を再表示するか／たたんで確定に進ませるか」のUI挙動。初期は**確定に進める（再表示は任意）**をたたき台とする。

### 10.6 タイムアウト／再試行の方針（一度だけの保証）

- 【決定】AI呼び出しには**タイムアウト**を設け、ぶら下がりを防ぐ。
- 【決定】再試行をしても**二重質問にしない**（同じ報告に質問ラウンドが2回走らないよう、9.3の状態で守る）。
- 【未決→12章】タイムアウト秒数、再試行回数（0回＝即fallbackも候補）。

### 10.7 失敗の可観測性（ログ・エラー種別の区別／鍵を出さない運用）

- 【決定】失敗時は**エラーの種別**（鍵無効／モデル名違い／ネット／レート制限／リクエスト不正／その他）を切り分けられるようログに残す。
- 【決定】**鍵（APIキー）の値はログにも画面にも出さない**。出すのは種別と要点のみ（昨日の運用と同じ）。
- 【未決→12章】ログの粒度・保管先・アラートの要否。

### 10.8 昨日の実環境知見（要約呼び出し失敗）から得た教訓の反映

- 【決定（教訓1）】**プロバイダ固有のスキーマ制約は共通スキーマを汚さずプロバイダ側で吸収する**。
  - 実例：Gemini の `response_schema` が `additionalProperties` を受け付けず 400 になった件。共通スキーマは変えず、プロバイダ実装側で除去して解決した（原則2）。本機能の新スキーマも同じ方針で扱う。
- 【決定（教訓2）】**元例外を握りつぶさず、種別が後から分かる形にする**。抽象化層で例外を包むときも、原因の種別を失わないようにする（昨日は包んだ先で種別が見えずデバッグに一手間かかった）。
- 【決定（教訓3）】**まず fallback で「報告は出せる」を担保**してから、AI品質を足す。AIは落ちうる前提で設計する。

## 12. 段階的実装の区切り（最終形 vs 最初の一歩）

この章は本ドキュメント内に散らした **【未決→12章】** の集約先でもある。

### 12.1 最初の一歩に含める範囲（最小で価値が出る形）

- 【決定】クライアント全体で**単一の粒度基準**（7.2）。
- 【決定】ルール検出による薄い項目の判定（スライス1）と、薄い項目への**質問文の生成（Gemini）**、**一度だけ**の追加質問。
- 【決定】回答は本文へ追記し要約を作り直す（5章）。
- 【決定】必須/任意の管理者設定（6.1）。
- 【決定】失敗時は追加質問を省いて確定できる degrade（10章）。

### 12.2 後続フェーズに送る範囲

- 【決定】**グループ/チーム別の基準**（7.1）と上書き層の作り込み。
- 【決定】定型質問の高度な運用（条件付き差し込み・優先順位の精緻化：4.2・4.3）。
- 【未決→12章】音声回答（STT連携）、無回答の可視化・評価指標利用、確定後の追記需要。

### 12.3 各段階の完了条件

- 【決定】**最初の一歩の完了条件**：「単一基準で、薄い項目に一度質問→本文追記→要約再生成→確定が通り、AI失敗時も追加質問を省いて確定できる」ことを満たせば、最初の一歩は完了とする。
- 【未決→12章】**後続フェーズ**（管理画面・チーム別基準など）の完了条件（受け入れ基準）は引き続き本文着手時に具体化する。

### 12.4 未決事項の集約（本章で必ず詰めるもの）

- 3.1 しきい値の具体値（空・極端に短いの基準）／7.4 対象カテゴリの仮初期値（どのカテゴリを対象にするか）
- 4.3 生成質問と定型質問の優先順位／4.4 質問数の上限
- 5.1 追記の体裁／5.3 確定後の追記需要の扱い
- 6.3 無回答の記録可否
- 7.3 基準の解決順／7.4 設定項目の型・既定値・UI
- 9.1〜9.3 データモデル（質問/回答の保存、基準の格納先、状態の持ち方）
- 10.3 質問生成失敗時の定型のみ縮退／10.5 再来訪UI／10.6 タイムアウト・再試行回数

### 12.5 実装スライスの順序（決定：機能ファースト）

- 【決定】**機能ファースト**で進める。設定はまず**単一の固定値**（保存場所に置いた1つの値）を読む形で**機能本体を先に作り、本物のGeminiで検証**する。管理者役割・管理画面（土台）は機能が固まってから次のスライスで足し、チーム別基準はさらに後続とする。
  - 理由：粒度のしきい値と「一度だけ質問」の使い心地は、実環境で動かさないと正解が分からない**未知**の部分なので先に潰す。管理者役割・管理画面は作り方が読めている**既知**の仕事なので後でよい。
  - 原則7は守る：設定が固定値でも、コードに分岐を埋め込まず**保存場所から読む**（後でグループ別へ広げても破綻しない＝7.3 の上書き層と整合）。

- 【決定】スライス順：
  1. **設定の置き場所（単一値）＋ルールの足切り**で「薄い項目」を判定する（質問はまだ出さない）。ルールの足切りは、**対象カテゴリ設定（仮の初期値）を読み、その対象カテゴリだけ**を空・極端に短いで判定する（3.1）。
  2. **薄い項目（スライス1で検出）への質問文の生成を Gemini で行う**（抽象化層の背後＝原則2、出力は固定スキーマ＝質問文＝原則3）＋ **degrade**（生成失敗時は質問なしで確定／10章）。
  3. **一度だけ質問を提示 → 回答 → 本文に追記 → 要約再生成**＋「質問済み」状態管理（9.3）。
  4. **必須/任意の確定ブロック**（6.2 の決定どおり）。→ ここで**本物のGeminiに報告を流し、しきい値と使い心地を調整**する。
  5. （その後＝土台）**管理者役割＋権限境界＋管理画面**で設定を編集可能にする（原則1）。ここに**対象カテゴリを付け替える管理画面**を含める（変更は翌日以降の報告に適用＝7.4）。
  6. （さらに後続）**チーム別の基準**（7.1・12.2）。
