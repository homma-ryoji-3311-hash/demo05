# 業務報告・スキルシート生成システム

要件定義 ／ 設計仕様書

客先常駐スタッフ向け 業務報告・要約・職務経歴自動生成基盤

Version 1.0（ドラフト）

作成日：2026年6月3日

# 1. はじめに

## 1.1 目的と背景

客先に常駐するスタッフの日々の業務報告（テキスト・音声）を蓄積し、蓄積データをAIで要約・専門用語化したうえで、指定のExcelフォーマット（職務経歴書／スキルシート）へ自動で書き込み・生成するWebシステムである。

常駐スタッフは日報の作成負担が大きく、また個々の業務実績が職務経歴として体系的に蓄積されにくい。本システムは、日々の報告を起点に、要約・実績の構造化・職務経歴への変換までを一気通貫で自動化し、スタッフ・管理者双方の負担軽減と実績の可視化を目的とする。

## 1.2 システムの位置づけ（4つの役割）

本システムは単なる日報ツールではなく、次の4つの役割を併せ持つ基盤として位置づける。

- **スタッフステータス管理：**誰がいつ報告したか、未報告は誰か、要確認事項は何かを把握する。

- **スキルシート生成システム：**蓄積した実績を職務経歴書／スキルシートへ自動変換・出力する。

- **業務報告自動要約：**テキスト・音声の報告を、インシデント・成果・課題などに自動で構造化・要約する。

- **スタッフ支援のためのコミュニケーション／マネジメントツール：**所感・課題・雑感（メンタル面）を通じて、現場の意識向上とスタッフのウェルビーイングを支える。

特に4つ目の役割は、管理が一方的に評価する道具ではなく、人が人をケアし、現場の改善と本人の成長を後押しする入口として設計する。

## 1.3 対象ユーザーと権限

| **権限** | **対象** | **主な操作** |
| --- | --- | --- |
| スタッフ | 客先常駐スタッフ | 業務報告の入力（自由文・設問・音声）、通知時刻の設定、自身のスキルシートの閲覧・ダウンロード |
| 営業・管理者 | 営業／管理担当 | 担当グループのスタッフ一覧閲覧、設問・Excelテンプレートの管理、生成済みExcelの個別／一括ダウンロード |

権限の境界（スタッフは自分のみ、管理者は担当グループ横断）は、画面表示の出し分けだけでなくバックエンド側で必ず強制する。

## 1.4 用語定義

| **用語** | **定義** |
| --- | --- |
| 生報告ログ | スタッフが確定した報告の不変の記録。確定後は書き換えない（再生成・監査の基礎）。 |
| マスター元データ | 生報告ログを案件単位で突合・集約した、常に最新状態の要約データ。スキルシート生成の元になる。 |
| 突合 | 新しい報告を既存のマスター元データにマージし、重複を増やさず状態を最新へ更新する処理。 |
| 役割タグ | 設問の答えがデータ上どの役割（案件・ステータス・スキル・内部のみ等）を担うかを宣言するタグ。 |
| グループ | 案件属性／職種の区分（例：エンジニア、マーケター、研修生）。設問やテンプレート、担当管理者の切り替え単位。 |
| アンカー | Excelテンプレート上で値の差し込み位置を示すプレースホルダや名前付き範囲。 |

# 2. システム全体像

## 2.1 全体アーキテクチャ

フロントエンドはPC／スマートフォン両対応のレスポンシブUI、バックエンドはAI処理・ファイル生成・スケジューリングを担うAPIサーバ、生成物はオブジェクトストレージに格納し、メタデータと署名付きURLをDBで管理する構成とする。認証はGoogle OAuth 2.0で行い、AI処理（要約・変換）は提供元を差し替え可能な抽象化層の背後に置く。

**重要な前提：**AIの要約モデルは認証（Google OAuth）や個人のGeminiプランに依存しない。OAuthは本人確認・認可のみを担い、要約処理は別系統のAPI（Claude／Gemini API／Vertex AI 等）を組織側で集約して呼び出す。

## 2.2 技術スタック（推奨）

| **領域** | **推奨** | **備考** |
| --- | --- | --- |
| フロントエンド | Next.js（React）＋レスポンシブCSS | PC／スマホを1コードベースで対応 |
| バックエンド | Python（FastAPI） | AI処理・Excel生成・非同期処理に適する |
| データベース | PostgreSQL | ユーザー・報告・案件・生成履歴等 |
| ファイル格納 | S3互換オブジェクトストレージ | 生成Excel・テンプレート・音声 |
| ジョブ実行 | Celery＋Redis または APScheduler | 通知のスタッフ別時刻判定等 |
| AI要約 | 提供元非依存の抽象化層 | Claude／Gemini API／Vertex AI を差し替え可能に |
| 音声テキスト化 | Web Speech API ／ サーバ側STT | Whisper・Deepgram・Google STT 等から選定 |
| Excel生成 | openpyxl | テンプレートのアンカーへ書き込み |

## 2.3 マルチグループ方針（一元化＋グループ別設定）

システムは1つに一元化し、デプロイもデータベースも単一とする。案件属性／職種は「グループ」という設定として持たせ、グループ固有の部分（報告の設問セット、スキルシート様式、担当管理者、タブ表示）だけを切り替える。全体機能（認証・入力・AI要約・突合・Excel生成・権限）は全グループ共通である。

この設定駆動の構造により、グループの追加・変更は設定の追加だけで完結し、コード改修を伴わない。

## 2.4 処理フロー（①〜⑤）

| **#** | **処理** | **内容** |
| --- | --- | --- |
| ① | 入力 | 自由文／設問回答／音声のいずれかで報告を入力。前回の入力と前回のAI要約を参照表示。 |
| ② | AI要約・下書き保存 | AIが内容を構造化（インシデント・成果・課題等）。人が確認・修正できる。 |
| ③ | 報告を確定 | 確認後に内容を確定して送信。 |
| ④ | 生報告ログに格納 | 確定内容を不変ログとして保持（再生成・監査の基礎）。 |
| ⑤ | 突合・再要約 | 既存マスターと新報告を案件単位でマージし、マスター元データを更新。 |

スキルシートはマスター元データを元に、別途生成する（第3.5節）。

# 3. 機能要件

## 3.1 認証・ユーザー・権限管理

- 認証方式はGoogle OAuth 2.0。法人GWSアカウントでのログインを必須とする。

- 社外ユーザーの混入を防ぐため、許可ドメインまたは招待制によるホワイトリスト管理を行う。

- ユーザーはロール（スタッフ／管理者）と所属グループを持つ。

- アクセス境界（スタッフは自分のみ、管理者は担当グループ横断）はバックエンドで強制する。

## 3.2 業務報告入力

- 入力方式は「自由文」「設問回答」「音声」から選択でき、最終的に編集可能なテキストへ集約する。

- 自由文モードではキーボード入力と音声入力を同一画面で併用でき、音声は本文へ追記される。

- 設問回答方式の設問はグループごとに管理者が定義する（第3.6節）。

- **前回入力の保持：**前回の本文／回答と前回のAI要約を参照表示し、起案コストの低減と入力品質の向上を図る。参照は読み取り専用で、丸写しを誘発しない控えめな表示とする。

- 入力は常時下書き自動保存し、通信断や離席でも内容を失わない。

## 3.3 音声処理・AI要約

**音声の扱い：**AIの要約モデルは音声を直接文字起こししない。音声はまずSTT（Web Speech APIまたはサーバ側STT）でテキスト化し、そのテキストを要約処理に渡す2段構成とする。

- STTの結果は確定前に必ずユーザーへ提示し、誤認識を修正できるようにする。

- **AI要約確認・編集画面：**AIが構造化した結果（対応案件・インシデント・成果・課題・抽出スキル等）を、確定前に人が確認・修正する。すべての項目を編集可能とする。

- 元の入力と要約を並べて照合できるようにする。

- 不確実・不足箇所には「要確認」フラグを付与し、ユーザーの確認を促す（例：定量指標の欠落）。

- 対応案件は確定時に既存案件へ紐づけ（案件キー）でき、これにより後段の突合を決定的に行える。

- インシデントは「発生／対応中／解決」のステータスを持ち、突合時の状態更新（上書き）に用いる。

## 3.4 突合・マスター元データ

- 生報告ログ（不変）と、突合済みのマスター元データを別レイヤーで保持する。

- **増分突合：**全履歴を毎回再処理せず、「既存マスター＋新報告」のみをマージして更新する（コスト・遅延・上限対策）。日次→月次→通算と段階的に畳み込む。

- **状態更新：**単純追記ではなく、同一案件キーで突合し、ステータス等は最新で上書きする（例：先週のインシデントを解決済みへ）。

- **実行タイミング：**報告確定のたびに同期実行する方針（本ワークロードは軽量でコストが小さいため）。運用規模により夜間バッチへ切り替え可能とする。

## 3.5 スキルシート（Excel）生成

生成は3フェーズに分離し、AIは言い換えのみを担い、データ取得と書き込みは決定的に行う。

| **フェーズ** | **内容** |
| --- | --- |
| データ組み立て | マスター元データ・案件・スキル集計をDBから決定的に取得する。 |
| AI変換 | 事実を職務経歴書向けの専門用語・客観的成果へ言い換える。マスターに無い数値は創作しない。出力はアンカー対応のJSONスキーマに固定。 |
| テンプレート反映 | openpyxlでアンカー位置へ書き込む。案件ブロックは雛形行を複製し、書式（罫線・結合・フォント）を維持する。 |

- **出力ファイル名：**「[スタッフ名]_[ファイル名]_YYYYMMDD.xlsx」をサーバ側で機械的に付与する（日付は出力日）。

- **格納と履歴：**生成物はオブジェクトストレージに格納し、DBにメタデータと署名付きURLを保持。再生成のたびに新ファイルを作り履歴として残す。

- **プレビュー：**xlsxはプレビュー用にPDFまたはHTMLへ変換して表示し、ダウンロードは元のxlsxを渡す。

## 3.6 設問テンプレート管理

- 管理者はグループ単位で設問セットを自由に作成・編集・並べ替えできる（標準機能）。

- 各設問は回答形式（短文／長文／選択）と必須／任意を指定できる。

- **役割タグ：**各設問に「案件キーに紐づけ」「ステータス付き」「スキル抽出」「内部のみ（シート非反映）」等の役割を宣言する。これにより自由な設問でも突合とシート生成を決定的に行える。

- **ガードレール：**公開時に必須の役割（案件キーが最低1つ、スキル抽出が最低1つ等）が揃っているか検証し、欠落時は警告する。

- **バージョン管理：**各報告は作成時のテンプレート版を保持し、編集は次回報告から反映する（下書き→公開でバージョンを切る）。過去報告は壊さない。

## 3.7 ソフト設問・ウェルビーイング

「雑感（メンタル面）」「所感（本人の振り返り）」「課題（現場意識の向上）」「AIを業務に活かせたこと」を設問として追加できる。答えの流れ先を役割で分ける。

| **設問** | **流れ先** | **扱い** |
| --- | --- | --- |
| AIを業務に活かせたこと | スキルシート | AI活用を実績・スキルとして職務経歴へ反映（スキル抽出）。 |
| 課題 | 内部の振り返り | 現場改善のための内部データ。原則シート非反映。 |
| 所感 | 内部の振り返り | 本人の成長記録。シート非反映。 |
| 雑感（メンタル面） | ウェルビーイング | HR・本人に限定。AI変換・シート反映から完全に除外。 |

**ウェルビーイングへの配慮：**雑感はAIの専門用語化・職務経歴変換の対象から完全に除外する。閲覧範囲はHR／担当者（と本人）に限定し、任意入力・本人による非公開を検討する。

- メンタル面をスコア化・ダッシュボード化して監視的に扱うことは避ける（正直さとウェルビーイングを損なうため）。

- AIは状態の診断・点数化を行わない。明らかに強い不調がうかがえる場合のみ、静かにHR／担当者へ人の対応を橋渡しする軽い検知に留める。

## 3.8 通知（リマインド）

- スタッフごとに個別のリマインド時刻を設定できる（例：毎日18:00）。

- **実装方式：**全員分のスケジュールを個別に立てず、短間隔のジョブが「今この時刻に通知すべきユーザー」をDBから抽出して送信する方式とする。

- タイムゾーンは保存をUTC、表示・判定をユーザーのローカル時刻で統一する。

- **通知チャンネル：**Slack（メイン、Webhook／Bot）。メールはオプションで設定によりON／OFF可能。

- 管理コンソールの「未報告」ステータスと連動し、誰にリマインドが必要かを可視化する。

## 3.9 管理者コンソール

- スタッフ一覧（PC前提）：氏名・客先・最終報告・状況（報告済み／未報告）・最新スキルシート・操作を一覧表示する。

- 各行の操作：プレビュー、最新スキルシートのダウンロード、再生成。未生成の場合はその旨を表示する。

- 「全員分を生成」：最新のマスター元データから全スタッフのシートをまとめて再生成する。

- 要確認の残件（確定済みだが数値未記載等）をフラグ表示する。

- 設問テンプレート編集、スキルシートテンプレート管理（アンカー検証付き）、一括ダウンロード（ZIP、客先・部署で絞り込み）。

- タブはグループに対応し、担当管理者には自分の担当グループのスタッフのみ表示する。

# 4. データモデル（主要エンティティ）

| **エンティティ** | **主な項目** | **役割** |
| --- | --- | --- |
| USERS | id, email, name, role, group_id, timezone | ユーザー・ロール・所属グループ |
| REPORTS（④） | id, user_id, report_date, input_mode, raw_text, confirmed_summary(JSON) | 確定報告の不変ログ |
| PROJECTS | id, user_id, project_key, client_name, status, last_active_at | 案件マスター（突合キー） |
| REPORT_PROJECTS | report_id, project_id, contribution | 報告と案件の紐づけ |
| INCIDENTS | id, project_id, status, opened_by, resolved_by | インシデントの状態管理 |
| SKILLS | id, name, category | 専門用語マスター（名寄せ） |
| REPORT_SKILLS | report_id, skill_id | 報告からのスキル抽出・在庫の素 |
| MASTER_SUMMARIES（⑤） | id, user_id, project_id, period, summary(JSON), reconciled_at | 突合済みマスター元データ |
| TEMPLATES | id, group_id, file_url, version, anchor_map | グループ別Excelテンプレート |
| QUESTION_SETS | id, group_id, version, questions(JSON＋役割タグ) | グループ別設問テンプレート |
| GENERATED_SHEETS | id, user_id, template_id, file_url, filename, generated_at | 生成済みスキルシート（履歴） |
| NOTIFICATION_SETTINGS | id, user_id, remind_time, slack_enabled, email_enabled | 通知設定 |

REPORTSは確定後不変、正規化層（PROJECTS／INCIDENTS／REPORT_SKILLS）は突合の土台、MASTER_SUMMARIESは⑤の成果でスキルシート生成の入力となる。

# 5. AI処理の設計方針

- **提供元非依存：**要約・変換は1つの差し替え可能なインターフェースの背後に置き、Claude／Gemini API／Vertex AIを切り替え可能にする。認証（GWS）は固定のまま、モデルだけ自由に選定できる。

- **構造化出力：**AIの出力はアンカー／フィールドに対応したJSONスキーマに固定し、後段の書き込み・突合を機械的に行う。

- **創作の禁止：**職務経歴の信頼性のため、マスターに無い数値・事実をAIに創作させない。不足は空欄のままとする。

- **コスト：**日報1件の要約は概ね1〜2千トークンで、安価なモデルなら1件あたり1円を大きく下回る。確定のたびの同期実行でも費用は小さい（単価は選定時に最新値で再確認）。

- **GWS環境内での完結：**クライアントの管理下で完結させたい場合は、クライアントのGCPプロジェクトでVertex AIを用い、Google組織内・組織課金とする選択肢を持つ。

# 6. 非機能要件

## 6.1 セキュリティ・権限

- Google OAuth 2.0による認証、許可ドメイン／招待制でのアクセス制御。

- ロール・グループに基づくアクセス境界をバックエンドで強制。

- 生成物・テンプレート・音声はオブジェクトストレージに安全に格納し、署名付きURLで配布。

## 6.2 プライバシー・守秘

- 報告には客先の機密情報が含まれるため、取り扱いは常駐先の守秘義務・社内ポリシーと整合させる。

- 管理者の閲覧範囲はグループ単位で限定する（担当外グループは閲覧不可）。

- 雑感（メンタル面）はHR・本人に限定し、AI変換・シート反映から除外する。

## 6.3 ユーザビリティ・その他

- PC／スマートフォン両対応のレスポンシブUI。スタッフ入力はモバイルファースト、管理者は一覧性重視のPC前提。

- 録音権限拒否・通信断・STT失敗時も入力内容を失わないエラーハンドリング。

- アクセシビリティ：状態は色だけでなくテキストでも提示、キーボードのみで操作完結。

- タイムゾーンは保存UTC・表示ローカルで統一。

# 8. 開発ロードマップ（段階導入）

価値の中心（テキスト報告→AI要約→Excel出力）を最初に縦に1本通し、その後に音声・通知・グループ別設定を拡張する。

| **フェーズ** | **範囲** | **ねらい** |
| --- | --- | --- |
| Phase 1 | 認証＋テキスト報告＋AI要約・確認画面 | 中核の入力〜要約を最短で通す |
| Phase 2 | Excelテンプレート＋スキルシート生成・保管・DL | 成果物の出力を確立 |
| Phase 3 | 突合・マスター元データ（⑤） | 蓄積と再要約の自動化 |
| Phase 4 | 通知（Slack）＋スタッフ別時刻 | 報告の定着を促進 |
| Phase 5 | 音声入力（Web Speech→必要に応じサーバSTT） | 入力手段の拡張 |
| Phase 6 | グループ別設定・設問エディタ・ウェルビーイング・管理者一括 | 多職種運用とケア機能 |

# 9. 今後の検討事項（決定待ち論点）

- スタッフは必ず1グループか、複数グループ所属を許すか。

- 管理者は1タブ専任か、複数タブ兼任を許すか。

- グループ移管時の過去報告・マスターデータの引き継ぎ方（履歴保持／表示タブ移管）。

- 雑感の閲覧範囲・任意／非公開の具体仕様。

- STT方式の選定（Web Speech API／サーバ側STT）と精度・モバイル対応。

- 要約モデルの選定（Claude／Gemini API／Vertex AI）と提供形態。

- 突合の実行タイミング（同期を基本としつつ、規模拡大時の夜間バッチ移行）。

- プレビュー方式（PDF変換／HTML変換）の選定。
